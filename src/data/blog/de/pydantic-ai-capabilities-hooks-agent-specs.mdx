Pydantic AI hat gerade die groesste API-Aenderung seit dem Launch veroeffentlicht. Capabilities, Hooks und Agent Specs sind in v1.71+ gelandet und veraendern grundlegend, wie man Agenten erweitert.

Wir pflegen 5 Open-Source-Bibliotheken, die auf [Pydantic AI](https://ai.pydantic.dev/) aufbauen: pydantic-ai-shields (frueher pydantic-ai-middleware), pydantic-ai-subagents, pydantic-ai-summarization, pydantic-ai-backend und das [Full-Stack AI Agent Template](https://github.com/vstorm-co/full-stack-fastapi-nextjs-llm-template). Alle fuenf wurden migriert. Dieser Artikel beschreibt was sich geaendert hat, warum es wichtig ist, und zeigt echten Vorher/Nachher-Code aus unseren Repositories.

## Was sind Capabilities?

Capabilities sind wiederverwendbare, komponierbare Einheiten von Agent-Verhalten. Anstatt mehrere Konfigurationsargumente einzeln zu uebergeben -- Tools hier, Instruktionen dort, Model-Settings woanders -- buendelt eine Capability alles in einen einzigen `capabilities`-Parameter des `Agent`-Konstruktors.

Jede Capability kann bereitstellen:

- **Tools** (ueber `get_toolset()`)
- **Instruktionen** (statische Strings oder dynamische Callables)
- **Model-Einstellungen** (Konfiguration pro Schritt)
- **Lifecycle Hooks** (before/after/wrap-Muster fuer Runs, Model-Requests, Tool-Aufrufe)
- **Tool-Vorbereitung** (Tool-Definitionen pro Schritt filtern oder modifizieren)

Die Basisklasse ist `AbstractCapability`. Man erbt davon, ueberschreibt die benoetigten Methoden und uebergibt Instanzen an den Agenten:

```python
from pydantic_ai import Agent
from pydantic_ai.capabilities import AbstractCapability

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[MyCapability(), AnotherCapability()],
)
```

Mehrere Capabilities komponieren automatisch. Before-Hooks feuern in Reihenfolge (cap1 dann cap2), After-Hooks umgekehrt (cap2 dann cap1), und Wrap-Hooks verschachteln sich als Middleware-Schichten.

## Was sind Hooks?

Hooks sind die Lifecycle-Abfangpunkte innerhalb von Capabilities. Pydantic AI bietet Hooks auf vier Ebenen:

1. **Run Hooks** -- `before_run`, `wrap_run`, `after_run`, `on_run_error`
2. **Node Hooks** -- `before_node_run`, `wrap_node_run`, `after_node_run`, `on_node_run_error`
3. **Model Request Hooks** -- `before_model_request`, `wrap_model_request`, `after_model_request`, `on_model_request_error`
4. **Tool Hooks** -- aufgeteilt in Validierung und Ausfuehrung, jeweils mit before/wrap/after/error-Varianten

Plus `prepare_tools` zum Filtern der Tool-Sichtbarkeit pro Schritt.

Das sind etwa 20 Hook-Punkte auf 4 Lifecycle-Ebenen. Error Hooks verwenden ein elegantes Muster: **raise um zu propagieren, return um wiederherzustellen**.

Fuer einfache Faelle bietet die `Hooks`-Capability Decorator-basierte Registrierung ohne Subclassing:

```python
from pydantic_ai.capabilities import Hooks

hooks = Hooks()

@hooks.on.before_model_request
async def log_request(ctx, request_context):
    print(f"Sende {len(request_context.messages)} Nachrichten")
    return request_context

agent = Agent("openai:gpt-4.1", capabilities=[hooks])
```

## Was sind Agent Specs?

Agent Specs trennen die Agent-Konfiguration vollstaendig vom Code. Man definiert seinen Agenten in YAML oder JSON:

```yaml
model: anthropic:claude-opus-4-6
instructions: You are a helpful assistant.
capabilities:
  - WebSearch
  - Thinking:
      effort: high
```

Dann laedt man ihn:

```python
agent = Agent.from_file("agent.yaml")
```

Capabilities, die `get_serialization_name()` und `from_spec()` implementieren, sind automatisch verfuegbar. Eigene Capabilities koennen also auch YAML-gesteuert sein.

## Wie unsere Bibliotheken migriert wurden

### pydantic-ai-middleware zu pydantic-ai-shields

Dies war die dramatischste Aenderung. Unsere Middleware-Bibliothek war gewachsen und enthielt `MiddlewareAgent`, `MiddlewareChain`, `ParallelMiddleware`, `ConditionalMiddleware`, `PipelineSpec`, Config-Loader, einen Compiler -- eine ganze parallele Abstraktionsschicht.

**Wir haben alles geloescht.**

Das v0.3.0-Release benannte das Paket in `pydantic-ai-shields` um und baute alles als Capabilities neu auf.

**Vorher (Middleware-Aera):**

```python
from pydantic_ai_middleware import MiddlewareAgent, CostTrackingMiddleware

middleware_agent = MiddlewareAgent(
    agent,
    middlewares=[CostTrackingMiddleware(budget_limit_usd=5.0)]
)
result = await middleware_agent.run("Hello")
```

**Nachher (Capabilities-Aera):**

```python
from pydantic_ai import Agent
from pydantic_ai_shields import CostTracking, PromptInjection, PiiDetector

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[
        CostTracking(budget_usd=5.0),
        PromptInjection(sensitivity="high"),
        PiiDetector(detect=["email", "ssn", "credit_card"]),
    ],
)
result = await agent.run("Hello")
```

Kein Wrapper-Agent. Keine Middleware-Kette. Die Shields sind einfach Capabilities, die sich in `before_run`, `after_run`, `prepare_tools` und `before_tool_execute` einklinken.

Das neue Paket liefert 10 Capabilities: `CostTracking`, `ToolGuard`, `InputGuard`, `OutputGuard`, `AsyncGuardrail` fuer Infrastruktur, plus `PromptInjection`, `PiiDetector`, `SecretRedaction`, `BlockedKeywords` und `NoRefusals` als Content-Shields ohne externe Abhaengigkeiten.

### pydantic-ai-subagents

Die Subagent-Bibliothek stellt jetzt `SubAgentCapability` bereit, das Toolset und dynamische Instruktionen buendelt:

```python
from pydantic_ai import Agent
from subagents_pydantic_ai import SubAgentCapability, SubAgentConfig

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[SubAgentCapability(
        subagents=[
            SubAgentConfig(
                name="researcher",
                description="Researches topics",
                instructions="You are a research assistant.",
            ),
        ],
    )],
)
```

### pydantic-ai-summarization

Vier Capabilities ersetzen das alte Middleware-basierte Kontextmanagement:

- **`SummarizationCapability`** -- loest LLM-Zusammenfassung bei Schwellenwertuebrschreitung aus
- **`SlidingWindowCapability`** -- kostenlose Alternative, die aelteste Nachrichten verwirft
- **`LimitWarnerCapability`** -- injiziert Warnungen wenn Limits sich naehern
- **`ContextManagerCapability`** -- Gesamtpaket: Token-Tracking, Auto-Komprimierung, Tool-Output-Kuerzung

```python
from pydantic_ai import Agent
from pydantic_ai_summarization.capability import ContextManagerCapability

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[ContextManagerCapability(max_tokens=100_000)],
)
```

### pydantic-ai-backend

Unser Dateisystem-Toolkit wurde zu `ConsoleCapability` -- buendelt Tools, Instruktionen und Berechtigungsdurchsetzung:

```python
from pydantic_ai import Agent
from pydantic_ai_backends import ConsoleCapability
from pydantic_ai_backends.permissions import READONLY_RULESET

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[ConsoleCapability(permissions=READONLY_RULESET)],
)
```

## Wichtigste Erkenntnisse

**1. Loescht eure Abstraktionsschichten.** Wir haben Tausende Zeilen Middleware-Code entfernt. Das Framework macht es besser.

**2. Komposition ist kostenlos.** Mehrere Capabilities stapeln sich ohne eigene Merge-Logik.

**3. Agent Specs veraendern das Deployment.** Agent-Verhalten in YAML definieren, durch Aenderung einer Config-Datei deployen.

**4. Die Migration ist mechanisch.** Fuer jeden Middleware-Hook gibt es ein direktes Capability-Aequivalent.

**5. Denkt in Capabilities, nicht in Agenten.** Altes Muster: spezialisierten Agent bauen. Neues Muster: Capability bauen, an jeden Agent anhaengen.
