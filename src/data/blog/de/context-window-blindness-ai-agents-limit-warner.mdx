> **TL;DR:** KI-Agenten haben keine intrinsische Wahrnehmung ihrer eigenen Kontextnutzung — diese Information liegt in der Orchestrierungsschicht, nicht in der Nachrichtenhistorie. `LimitWarnerCapability` in pydantic-deep v0.3.8 überbrückt diese Lücke, indem es Benutzernachrichten bei 70% (URGENT) und 85% (CRITICAL) injiziert und dem Modell Zeit gibt, die Arbeit elegant abzuschließen, bevor die automatische Kompression bei 90% einsetzt. Ebenfalls in v0.3.8: BM25-basierte Historiensuche ohne externe Abhängigkeiten, `EvictionCapability` die große Tool-Ausgaben abfängt bevor sie in die Historie gelangen, und erweiterte Kontextdatei-Erkennung (7 Dateitypen).

Am Montag habe ich gezeigt, wie Agenten Token verschwenden, indem sie in Schleifen steckenbleiben. Heute — ein stilleres Problem, das genauso viel kostet und weitaus schwieriger zu erkennen ist.

Dein KI-Agent ist erblindet. Er hatte keine Ahnung, dass das Kontextfenster zu 90% voll war.

## Das Problem: Zwei verschiedene Realitäten

Wenn du eine lange Agentenaufgabe ausführst, siehst du: eine Statusleiste, die "Kontext: 87% genutzt" zeigt. Es ist dort in der TUI. Du siehst, dass dem Agenten fast kein Platz mehr bleibt.

Aber das Modell kann die Statusleiste nicht sehen. Es hat keine Ahnung.

Aus der Perspektive des Modells läuft alles normal — jede geschriebene Nachricht, jeder Tool-Aufruf, jeder Plan. Es gibt kein Signal, dass sich die Konversationshistorie füllt. Es generiert weiterhin lange Antworten, initiiert mehrstufige Pläne, ruft Tools auf.

Dann bei 90%: automatische Kompression. Das Arbeitsgedächtnis des Modells wird zwangskomprimiert. Es verliert den Faden dessen, was es 40 Nachrichten zuvor getan hat. Es beginnt, frühere Entscheidungen zu widersprechen.

Das ist Kontextfenster-Blindheit: die Lücke zwischen dem, was der Benutzer sieht, und dem, was das Modell über seine eigene Situation weiß.

## Warum das passiert

Moderne LLM-Interfaces zeigen die Kontextnutzung in der UI — für den menschlichen Operator. Aber das Modell selbst sieht nur die Nachrichtenhistorie, ohne intrinsisches Bewusstsein dafür, wie viel Kontextfenster verbraucht wird.

## Die Lösung: LimitWarnerCapability

In [pydantic-deep v0.3.8](https://github.com/vstorm-co/pydantic-deep) haben wir `LimitWarnerCapability` hinzugefügt. Die Lösung ist einfach: Injiziere die Kontextnutzungsinformation direkt in die Konversation als Benutzernachricht, bei zwei Schwellenwerten.

**Bei ~70% Nutzung (URGENT):**

```
You are approaching the context limit. You have used approximately 70% of your available context window. Begin wrapping up your current task. Avoid starting new complex subtasks.
```

**Bei ~85% Nutzung (CRITICAL):**

```
CRITICAL: Your context window is almost full. Stop current work and use /compact NOW to compress history before continuing. Do not start new tasks.
```

Dies sind keine System-Prompt-Modifikationen. Sie werden als Benutzernachrichten injiziert — der Typ, den das Modell als autoritative Eingabe behandelt.

Die Fähigkeit ist automatisch aktiviert, wenn `context_manager=True` gesetzt ist (Standard in pydantic-deep):

```python
from pydantic_deep import DeepAgent

agent = DeepAgent(
    model="claude-opus-4-5",
    context_manager=True,  # Standard — aktiviert LimitWarnerCapability
)
```

## Warum nicht System-Prompt?

System-Prompt-Anweisungen ("wenn du merkst, dass dir der Kontext ausgeht, schließe die Aufgabe ab") funktionieren hier nicht gut — das Modell kann seine eigene Token-Nutzung nicht zuverlässig aus der Konversation heraus schätzen.

Laufzeit-Injektion bei spezifischen Schwellenwerten funktioniert, weil sie echte Daten aus der Orchestrierungsschicht verwendet.

## BM25-Historiensuche: Null Abhängigkeiten, reines Python

Wir haben auch `search_conversation_history` neu geschrieben:

```python
# Vor v0.3.8: naive Teilstring-Suche
results = [msg for msg in history if query in msg.content]

# Nach v0.3.8: BM25-gereiht
results = search_conversation_history(history, "explain the authentication flow")
# Seltene Terme (hoher IDF) ranken höher
# Mehrwort-Queries tokenisiert und einzeln bewertet
# Null externe Abhängigkeiten — reines Python
```

## EvictionCapability

`EvictionCapability` fängt große Tool-Ausgaben über den `after_tool_execute`-Hook ab — **bevor** sie in die Nachrichtenhistorie gelangen. Du preventest Kontext-Aufblähung statt darauf zu reagieren.

## Schlüsselerkenntnisse

- **Kontextfenster-Blindheit** ist real: Modelle haben keine intrinsische Wahrnehmung ihrer eigenen Kontextnutzung.
- **LimitWarnerCapability** injiziert Benutzernachrichten bei 70% und 85% — vor der automatischen Kompression bei 90%.
- **BM25-Suche** ersetzt naive Teilstring-Suche, null externe Abhängigkeiten.
- **EvictionCapability** fängt große Tool-Ausgaben vor dem Eintritt in die Historie ab.
- Alles automatisch mit `context_manager=True` (Standard).
