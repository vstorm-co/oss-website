> **TL;DR:** pydantic-deep v0.3.5 bringt `--sandbox docker` — ein Flag, das die Agenten-Oberfläche im Terminal belässt, während alle Dateioperationen und Shell-Befehle innerhalb eines Docker-Containers ausgeführt werden, mit dem Projekt eingehängt unter `/workspace`. Benannte Workspaces (`--workspace <name>`) bieten persistente Docker-Umgebungen sitzungsübergreifend. Der Headless-Runner (`pydantic-deep run`) fügt `--json`, `--max-turns` und `--timeout` für die CI/CD-Integration hinzu. Kein Dockerfile nötig. Container wird beim Beenden automatisch bereinigt.

Am Mittwoch haben wir die Installation auf einen einzigen Befehl reduziert. Jetzt sprechen wir darüber, wo der Agent tatsächlich läuft.

Wenn Sie einen KI-Agenten lokal ausführen, hat er vollen Zugriff auf die Host-Umgebung. Er kann Dateien ändern, Pakete installieren, Systemkonfigurationen verändern. In der Entwicklung ist das in Ordnung — Sie wollen, dass der Agent Zugriff hat. Aber in CI/CD oder beim Ausführen nicht vertrauenswürdiger Aufgaben ist das ein Problem.

pydantic-deep v0.3.5 — die modulare Agent-Runtime für Python — bringt den Docker-Sandbox-Modus. Hier ist, was wir gebaut haben, warum wir es so gebaut haben und wie man es in einer echten CI/CD-Pipeline einsetzt.

## Warum isolierte Ausführung wichtig ist

Das Risiko mit Agenten auf dem Host ist subtil. Es geht nicht darum, dass der Agent etwas Böswilliges tut — sondern dass er Dinge tun kann, die Sie nicht beabsichtigt haben. Ein Paket installieren, das mit Ihren Projektabhängigkeiten in Konflikt gerät. Eine Konfigurationsdatei beim Debuggen ändern. Zustand hinterlassen, der den nächsten Pipeline-Lauf bricht.

Die Lösung ist Isolation. Der Agent sollte das Projekt sehen, aber seine Seiteneffekte sollten nicht auf den Host auslecken.

## Docker-Sandbox-Architektur

In pydantic-deep ändert das Flag `--sandbox docker`, wo die Ausführung stattfindet, nicht wo die Oberfläche lebt.

- TUI und Headless-Runner bleiben wie gewohnt im Terminal
- Dateioperationen (`read_file`, `write_file`, `list_dir`) werden innerhalb des Docker-Containers ausgeführt
- Shell-Befehle (`run_command`) werden innerhalb des Containers ausgeführt
- Ihr Projektverzeichnis ist unter `/workspace` eingehängt (Lesen-Schreiben)

```bash
pydantic-deep run "analyze this codebase and suggest improvements" \
  --sandbox docker
```

Kein Dockerfile zu schreiben. Kein manuelles `docker run`. Konfigurieren Sie das Sandbox-Image in `.pydantic-deep/config.toml`:

```toml
[sandbox]
sandbox = "docker"
sandbox_image = "python:3.12-slim"
```

## Benannte Workspaces: Persistente Docker-Umgebungen

Benannte Workspaces lösen das Zustandsproblem — innerhalb installierte Pakete überleben zwischen Sitzungen:

```bash
pydantic-deep tui --workspace ml-env
# installierte Pakete überleben zwischen Sitzungen
# mehrere Threads teilen denselben Workspace
```

`--workspace <name>` impliziert automatisch `--sandbox docker`, Sie brauchen nicht beide Flags.

Workspace-Verwaltung:

```bash
# Alle Workspaces mit Status, Image, Erstellungsdatum
pydantic-deep sandbox list

# Bestimmten Workspace stoppen
pydantic-deep sandbox stop ml-env

# Alle Projekt-Workspaces stoppen und entfernen
pydantic-deep sandbox stop all --rm
```

## Headless-Runner für CI/CD

```bash
pydantic-deep run "run all tests and fix failures" \
  --sandbox docker \
  --max-turns 50 \
  --timeout 300 \
  --json
```

Wichtige CI/CD-Flags:

| Flag | Zweck |
|------|-------|
| `--json` | Maschinenlesbarer Output für Pipeline-Parsing |
| `--max-turns` | Iterationslimit des Agenten (verhindert unkontrollierte Kosten) |
| `--timeout` | Wall-Clock-Timeout in Sekunden |
| `--model` | Modell-Override pro Ausführung |
| `--no-subagents` | Deaktiviert Subagenten für deterministische Läufe |

pydantic-deep entfernt pydantic-ais Standard-50-Request-Limit, damit lange CI-Aufgaben nicht stillschweigend fehlschlagen. Sie kontrollieren Limits explizit über `--max-turns` und `--timeout`.

## Anforderungen

```bash
pip install 'pydantic-ai-backend[docker]'
```

## Wichtigste Erkenntnisse

- `--sandbox docker` führt Datei- und Shell-Operationen innerhalb eines Docker-Containers aus, mit dem Projekt unter `/workspace`
- Benannte Workspaces (`--workspace <name>`) bewahren installierte Pakete und Zustand zwischen Sitzungen
- Der Headless-Runner (`pydantic-deep run`) ist CI/CD-nativ: `--json`, `--max-turns`, `--timeout`, vollständige Flag-Parität mit TUI
- Workspace-Verwaltung (`pydantic-deep sandbox list/stop`) bietet Sichtbarkeit und Kontrolle
- Erfordert `pip install 'pydantic-ai-backend[docker]'`
