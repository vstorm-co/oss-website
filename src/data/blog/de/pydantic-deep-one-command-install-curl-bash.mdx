> **TL;DR:** Die Standard-Python-CLI-Tool-Installation umfasst 7 Schritte und 15+ Minuten: Python-Version, venv, pip install, falsche Extras, PATH-Verwirrung. pydantic-deep v0.3.6 löst das mit einem Ein-Befehl-Installer: `curl -fsSL https://raw.githubusercontent.com/vstorm-co/pydantic-deep/main/install.sh | bash`. Das Skript erkennt uv automatisch, installiert es bei Bedarf, führt `uv tool install "pydantic-deep[cli]"` aus und verifiziert PATH. `pydantic-deep update` übernimmt Upgrades. Startup-Update-Benachrichtigungen prüfen PyPI mit 2-Sekunden-Timeout und 24-Stunden-Cache — blockieren den Start nie.

So installiert man heute die meisten Python-KI-Tools:

1. Python installieren. (Welche Version? 3.10? 3.11? 3.12? README überprüfen. README sagt "Python 3.8+." Das hilft nicht.)
2. Virtuelle Umgebung erstellen.
3. Aktivieren. (Jedes Mal.)
4. `pip install tool-name` — auf Abhängigkeiten warten.
5. Tool starten. `ModuleNotFoundError: No module named 'textual'` erhalten.
6. `pip install tool-name[cli]` — die Extras waren in einer separaten Klammer.
7. Erneut versuchen. Jetzt funktioniert es.

Sieben Schritte. Mindestens fünfzehn Minuten. Für einen ersten Eindruck.

Das ist die Standard-Python-CLI-Installations-Erfahrung, und sie ist eine echte Hürde.

Wir haben das für [pydantic-deep](https://github.com/vstorm-co/pydantic-deep) behoben — the modular agent runtime for Python.

## Ein Befehl

```bash
curl -fsSL https://raw.githubusercontent.com/vstorm-co/pydantic-deep/main/install.sh | bash
```

Das ist die gesamte Installation. Kein Python-Versionsmanagement. Keine virtuellen Umgebungen. Keine Extras-Klammern.

## Was install.sh tatsächlich tut

**Schritt 1: uv erkennen**

[`uv`](https://github.com/astral-sh/uv) ist ein Python-Paketmanager von Astral, der virtuelle Umgebungen, Tool-Installationen und Python-Versionsmanagement transparent handhabt. Wenn uv vorhanden ist, verwendet das Skript es sofort. Wenn nicht, installiert es uv über den offiziellen Astral-Installer.

```bash
if command -v uv &> /dev/null; then
    echo "uv found, using existing installation"
else
    echo "Installing uv..."
    curl -LsSf https://astral.sh/uv/install.sh | sh
fi
```

**Schritt 2: pydantic-deep als uv-Tool installieren**

```bash
uv tool install "pydantic-deep[cli]"
```

`uv tool install` erstellt eine isolierte Umgebung für das Tool und macht die Binärdatei global verfügbar. Keine Aktivierung nötig. Die `[cli]`-Extras-Gruppe enthält alles für die vollständige TUI — einschließlich `textual`.

**Schritt 3: Verifizieren und Anleitung geben**

Das Skript führt `pydantic-deep --version` zur Verifikation aus. Wenn die Binärdatei nicht im PATH ist, gibt es genau aus, was zur Shell-Konfiguration hinzugefügt werden muss.

## Der textual-Bug, der alles startete

Vor v0.3.6 war die Standardinstallation `pip install pydantic-deep`. Dann begannen `ModuleNotFoundError: No module named 'textual'`-Berichte von frischen Installationen.

Das Problem: `textual` war in der `tui`-Extras-Gruppe, aber die meisten Benutzer gaben keine Extras an. Wir haben `textual` in die `cli`-Extras-Gruppe verschoben.

## Selbst-Update: `pydantic-deep update`

Das Update ist:

```bash
pydantic-deep update
```

Das ist `brew upgrade` für KI-Agenten. Darunter:

- Wenn uv verfügbar: führt `uv tool upgrade pydantic-deep` aus
- Wenn nicht: Fallback zu `pip install --upgrade "pydantic-deep[cli]"`

## Startup-Update-Benachrichtigungen

Jedes Mal wenn pydantic-deep gestartet wird, prüft es still PyPI:

```
pydantic-deep v0.3.6
Update available: v0.3.6 → v0.3.7  Run: pydantic-deep update
```

- 2-Sekunden-Timeout — blockiert den Start nie
- Ergebnisse in `~/.pydantic-deep/update_check.json` für 24 Stunden gecacht
- Implementiert mit `urllib` und threading — null zusätzliche Abhängigkeiten

## Warum uv?

Wir haben pipx (langsamer, eigene Installation nötig), Homebrew (Wartungsaufwand), native Binärdateien (zu fragil wegen dynamischer Imports) und uv evaluiert.

uv gewählt wegen: Geschwindigkeit, `uv tool` genau für diesen Anwendungsfall konzipiert, transparentes Python-Versionsmanagement.

## Schlüsselerkenntnisse

- **Die Standard-Python-CLI-Installation ist eine Hürde** — venv-Management, PATH-Probleme, falsche Extras.
- **install.sh übernimmt alles**: uv-Erkennung, Installation bei Bedarf, `pydantic-deep[cli]`, PATH-Verifizierung.
- **`pydantic-deep update`** ist ein Befehl zum Aktuell-Bleiben.
- **Startup-Benachrichtigungen** prüfen PyPI passiv mit 2-Sekunden-Timeout und 24-Stunden-Cache.
- **textual** ist jetzt korrekt in `cli`-Extras.
