## Das Problem: Generische Agenten in spezifischen Codebasen

Jede Codebasis hat Konventionen. Namensmuster. Fehlerbehandlungsansatze. Testorganisation. Import-Reihenfolge. Diese Konventionen existieren in den Kopfen der Entwickler, in README-Dateien die niemand liest, oder in Style-Guides die aus dem Takt geraten.

Wenn ein AI-Coding-Agent Ihrem Projekt beitritt, startet er bei null. Er folgt vielleicht allgemeinen Python-Best-Practices, aber er kennt nicht *Ihre* Best Practices.

Das Ergebnis: Sie verbringen die Halfte Ihrer Zeit damit, die Ausgabe des Agenten zu korrigieren, anstatt Features zu bauen.

Die meisten Teams losen das durch:
1. Konventionen in jeden Prompt einfugen (verschwendet Kontextfenster)
2. Eine `AGENTS.md`-Datei erstellen (besser, aber begrenzt)
3. `.cursorrules` verwenden (Cursor-spezifisch)
4. Uberhaupt nicht konfigurieren (am haufigsten)

Wir wollten etwas Strukturierteres. Etwas, das vom ersten Tag an mit dem Projekt ausgeliefert wird.

## Was in .claude/ enthalten ist

In [full-stack-ai-agent-template](https://github.com/vstorm-co/full-stack-ai-agent-template) v0.2.3 (veroffentlicht am 5. April) enthalt jedes generierte Projekt ein vollstandiges `.claude/`-Verzeichnis. Hier ist, was drin ist.

### settings.json — Berechtigungskonfiguration

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(git status)",
      "Bash(git log*)",
      "Bash(git diff*)",
      "Bash(git branch*)",
      "Bash(git checkout*)",
      "Bash(pytest*)",
      "Bash(ruff*)",
      "Bash(ty*)",
      "Bash(alembic*)"
    ]
  }
}
```

Das sagt Claude Code, was es ohne Nachfrage tun darf. Dateizugriff, Suche, Git-Operationen, Tests ausfuhren, Linting mit ruff, Typprufung mit ty (Astrals neuer Rust-basierter Type-Checker) und Datenbankmigrationen mit Alembic. Sichere Operationen, die ein Entwickler standig ausfuhrt.

Kein `Bash(rm*)`. Kein `Bash(docker*)`. Kein beliebiger Shell-Zugriff. Der Agent kann erkunden und validieren, aber nichts kaputt machen.

### 8 Regeldateien

Das `.claude/rules/`-Verzeichnis enthalt 8 Markdown-Dateien, die dem Agenten Ihre Projektkonventionen beibringen:

**architecture.md** — Projektstruktur. Wo Modelle leben (`app/models/`), wo API-Routen hingehen (`app/api/`), wie die Agent-Schicht sich mit dem WebSocket-Handler verbindet.

**code-style.md** — Formatierung und Benennung. Zeilenlangenlimits, Import-Reihenfolge (stdlib → Drittanbieter → lokal), `snake_case` fur Funktionen, `PascalCase` fur Klassen. Dinge die ruff erzwingt, aber erklart mit *warum*, damit der Agent korrekten Code beim ersten Versuch schreibt.

**schemas.md** — Pydantic-Modell-Konventionen. Basisklassen zum Erben, Feldbenennung, Validator-Muster, die Unterscheidung zwischen Request-Schemas, Response-Schemas und Datenbankmodellen.

**exceptions.md** — Fehlerbehandlung. Die benutzerdefinierte Exception-Hierarchie, wann `HTTPException` vs. Domain-Exceptions geworfen werden, Fehlerantwortformat.

**api-conventions.md** — FastAPI-Muster. Routenbenennung, Dependency-Injection-Nutzung, Response-Model-Deklarationen, Paginierungsmuster.

**testing.md** — pytest-Konventionen. Fixture-Organisation, Factory-Muster, Async-Test-Setup, was gemockt wird und was gegen echte Services getestet wird. Benennung: `test_<feature>_<scenario>_<expected>`.

**frontend.md** — Next.js- und React-Konventionen. Komponentenstruktur, State-Management-Muster, Tailwind v4-Nutzung, API-Client-Muster.

**general.md** — Querschnittsthemen. Umgebungsvariablen-Handhabung, Logging-Muster, Konfigurationsmanagement, Dokumentationsstandards.

### 3 Slash-Befehle

Das `.claude/commands/`-Verzeichnis bietet projektspezifische Workflows:

**`/project:review`** — Code-Review-Workflow. Der Agent pruft geanderte Dateien gegen alle Regeldateien, fuhrt ruff und ty aus, fuhrt relevante Tests aus und erstellt ein strukturiertes Review mit Schweregraden.

**`/project:add-endpoint`** — Neuen API-Endpoint erstellen. Erstellt Route, Schema, Service-Schicht und Testdatei — alles gemaess den Konventionen aus den Regeldateien. Ein Befehl, vier Dateien, null Korrekturen notig.

**`/project:fix-issue`** — Bugfix-Workflow. Nimmt eine Problembeschreibung entgegen, findet relevanten Code, schlagt einen Fix vor, schreibt Tests und validiert mit der bestehenden Testsuite.

## Warum nicht einfach AGENTS.md?

`AGENTS.md` (oder `CLAUDE.md` im Repository-Root) funktioniert. Wir nutzen es auch. Aber es gibt Kompromisse:

**Eine Datei vs. modulare Regeln.** Eine massive Markdown-Datei wird unhandlich. Acht fokussierte Dateien bedeuten, dass der Agent nur relevanten Kontext ladt.

**Kein Berechtigungsmodell.** `AGENTS.md` kann beschreiben was der Agent tun soll, aber nicht einschranken was er *kann*. `settings.json` bietet echte Leitplanken.

**Keine Workflows.** Sie konnen Prozesse in Markdown beschreiben, aber Slash-Befehle machen sie ausfuhrbar. `/project:add-endpoint` ist schneller als "bitte folgen Sie dem Endpoint-Erstellungsprozess beschrieben in AGENTS.md Abschnitt 4.2."

**Komplementar, nicht konkurrierend.** Wir verwenden weiterhin ein Top-Level `CLAUDE.md` fur High-Level-Projektkontext. Das `.claude/`-Verzeichnis behandelt die detaillierte, strukturierte Konfiguration. Sie arbeiten zusammen.

### Vergleich mit .cursorrules

`.cursorrules` dient einem ahnlichen Zweck fur Cursor. Die wichtigsten Unterschiede:

- `.cursorrules` ist eine einzelne Datei; `.claude/` ist ein Verzeichnis mit modularen Dateien
- `.cursorrules` hat kein Berechtigungsmodell
- `.cursorrules` hat kein Befehlssystem
- `.claude/` ist spezifisch fur Claude Code (vorerst — die Struktur ist portabel)

## Wichtigste Erkenntnisse

1. **AI-Coding-Agenten sind nur so gut wie ihr Kontext.** Generische Agenten produzieren generischen Code. Projektspezifische Regeln machen sie produktiv vom ersten Prompt an.
2. **Liefern Sie die Konfiguration mit dem Projekt aus.** Verlassen Sie sich nicht darauf, dass Entwickler die Agent-Konfiguration manuell einrichten.
3. **Modular schlagt monolithisch.** Acht fokussierte Regeldateien > ein 2000-Zeilen-Markdown-Dokument.
4. **Berechtigungen zahlen.** Dem Agenten zu erlauben Tests und Linter frei auszufuhren wahrend destruktive Operationen eingeschrankt werden, ist die richtige Balance.
5. **Slash-Befehle verwandeln Dokumentation in Workflows.** Ausfuhrbare Konventionen > Konventionen von denen Sie hoffen dass jemand sie liest.
