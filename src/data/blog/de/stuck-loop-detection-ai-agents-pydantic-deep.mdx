> **TL;DR:** Die meisten Agenten-Schleifen sind keine Modell-Fehler — sie sind mechanische Wiederholungen, die das Modell selbst nicht erkennt. pydantic-deep v0.3.8 führt StuckLoopDetection ein, eine Capability, die drei Schleifenmuster erkennt (wiederholte identische Aufrufe, A-B-A-B-Wechsel, No-Op-Schleifen), bevor sie Token verschwenden. Konfigurierbarer Schwellenwert, konfigurierbare Aktion (ModelRetry oder StuckLoopError), Per-Run-Zustandsisolation für parallele Runs. Standardmäßig aktiviert.

Am Sonntag veröffentlichte ich einen Überblick über fünf Releases, die wir in zwei Wochen geliefert haben. Heute beginne ich mit den Deep Dives. Zuerst: StuckLoopDetection — die Capability, die in v0.3.8 erschien und die wir Monate früher hätten bauen sollen.

Hier ist der Vorfall, der es offensichtlich machte.

Ein Coding-Agent arbeitete über Nacht an einer Refactoring-Aufgabe. Die Aufgabe: Importe eines Moduls umstrukturieren und abhängige Dateien aktualisieren. Der Agent kam halbwegs durch, stieß auf eine Datei mit einem ungewöhnlichen Import-Muster, versuchte sie zu lesen, konnte das Ergebnis nicht korrekt parsen, und fiel darauf zurück, die Datei erneut zu lesen.

Morgens: 47 Aufrufe von `read_file` auf demselben Pfad. $12 API-Kosten. Null Fortschritt bei der eigentlichen Aufgabe.

Das Modell war nicht "kaputt." Jeder Aufruf sah lokal vernünftig aus. Von außen: feststeckend.

---

## Warum Prompting nicht ausreicht

Die naheliegende Antwort auf Agenten-Schleifen sind bessere System-Prompts. "Wiederhole keine Tool-Aufrufe." "Wenn du dasselbe Ergebnis zweimal bekommst, versuche einen anderen Ansatz."

Das funktioniert manchmal. Die Probleme:

1. **Das Modell erkennt Schleifen oft nicht als Schleifen.** Jeder wiederholte Aufruf erscheint lokal gerechtfertigt. Der Agent "ignoriert" die Anweisung nicht — er sieht das Muster über Aufrufe hinweg wirklich nicht.
2. **Prompt-Compliance verschlechtert sich unter kognitiver Belastung.** Lang laufende Aufgaben mit vielen Tool-Aufrufen, komplexen Kontexten und mehreren Einschränkungen drängen das Modell dazu, Randfälle-Anweisungen zu ignorieren.
3. **Du musst die Anweisung jedem Agenten einzeln hinzufügen.** Es gibt keinen Standardplatz für "Schleifen-Prävention"-Logik, die überall gilt.

Erkennung auf Capability-Ebene behebt alle drei. Es bittet das Modell nicht, sich selbst zu überwachen — es überwacht von außen.

## Die drei Schleifenmuster

### Muster 1: Wiederholte identische Aufrufe

Das einfachste Muster. Der Agent ruft dasselbe Tool mit denselben Argumenten N-mal hintereinander auf.

```
Turn 12: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
Turn 13: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
Turn 14: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
```

**Warum das passiert:** Der Agent kann das Ergebnis nicht verarbeiten und hat kein anderes Tool als Fallback. Das Standardverhalten: dasselbe erneut versuchen.

**Standard-Schwellenwert:** 3 identische Aufrufe. Nach dem dritten löst Intervention aus.

### Muster 2: A-B-A-B-Wechsel-Schleifen

Subtiler. Der Agent wechselt zwischen zwei Tools, jedes gibt dem anderen ein unvollständiges Ergebnis.

```
Turn 8:  list_directory(path="src/")
Turn 9:  read_file(path="src/main.py")
Turn 10: list_directory(path="src/")
Turn 11: read_file(path="src/main.py")
Turn 12: list_directory(path="src/")
```

**Warum das passiert:** Tool A gibt etwas zurück, das Tool B nahelegt, Tool B gibt etwas zurück, das Tool A nahelegt, und der Agent kreist unendlich. Keines der Tools gibt dem Agenten genug, um voranzukommen.

Dieses Muster ist besonders teuer, weil es wie Fortschritt aussieht — der Agent ruft verschiedene Tools auf, wechselt den Kontext — aber kommt nirgendwo an.

### Muster 3: No-Op-Schleifen

Der Agent macht denselben Aufruf und bekommt dasselbe Ergebnis, macht aber trotzdem weiter.

```
Turn 5:  write_file(path="output.txt", content="result: error")  → success
Turn 6:  write_file(path="output.txt", content="result: error")  → success
Turn 7:  write_file(path="output.txt", content="result: error")  → success
```

**Warum das passiert:** Der Agent versucht ein Ergebnis zu "committen", merkt aber nicht, dass der Schreibvorgang bereits erfolgt ist.

## Die Implementierung

```python
from pydantic_deep import create_deep_agent
from pydantic_deep.capabilities import StuckLoopDetection

# Einfach: Standard verwenden
agent = create_deep_agent(
    model="anthropic:claude-opus-4-6",
    stuck_loop_detection=True,  # Standard: True, Schwellenwert: 3
)

# Angepasst: Schwellenwert und Aktion konfigurieren
agent = create_deep_agent(
    model="anthropic:claude-opus-4-6",
    capabilities=[
        StuckLoopDetection(
            max_repeated=5,
            action="warn",     # "warn" = ModelRetry, "error" = StuckLoopError
        )
    ],
)
```

**`action="warn"` (Standard):** Löst `ModelRetry` aus. Das Modell bekommt eine Nachricht und pivotiert meistens zu einem anderen Tool.

**`action="error"`:** Löst `StuckLoopError` aus. Nützlich für automatische Pipelines, die sauber scheitern wollen.

```python
from pydantic_deep.capabilities import StuckLoopDetection, StuckLoopError

try:
    result = await agent.run("refactor the imports in src/")
except StuckLoopError as e:
    print(f"Agent feststeckend: {e.pattern}-Muster erkannt")
```

### Per-Run-Zustandsisolation

Parallele `agent.run()`-Aufrufe teilen keine Erkennungs-Zustände:

```python
# Sicher, mit einer gemeinsamen Agenten-Instanz parallel auszuführen
results = await asyncio.gather(
    agent.run("analyze src/module_a.py"),
    agent.run("analyze src/module_b.py"),
)
```

Wird automatisch über `for_run()` gehandhabt.

## Der Business-Case

Eine 47-Aufruf-Schleife bei Claude Opus-Preisen kostet ~$12 für eine Aufgabe, die $0,50 hätte kosten sollen.

Kosten für `stuck_loop_detection=True`: Null API-Aufrufe, vernachlässigte Latenz, standardmäßig aktiviert. Selbst Falsch-Positive sind billig: ein ModelRetry, dann versucht der Agent einen anderen Ansatz.

---

## Wichtigste Erkenntnisse

- **Agenten-Schleifen sind meist mechanisch, keine Modell-Fehler.** Das Modell erkennt das Muster nicht; Erkennung muss außerhalb des Modells stattfinden.
- **Drei Muster** mit unterschiedlichen Grundursachen.
- **Erkennung auf Capability-Ebene** gilt einheitlich ohne per-Agenten-Prompt-Engineering.
- **Per-Run-Isolation** (`for_run()`) macht es sicher für parallele `agent.run()`-Aufrufe.
- **Standardmäßig aktiviert** — kein Konfigurationsaufwand erforderlich.
