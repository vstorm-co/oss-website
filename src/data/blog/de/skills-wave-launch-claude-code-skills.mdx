78% der Fortune-500-Unternehmen führen AI-Coding-Assistenten ein. 45% des generierten Codes geht mit Sicherheitslücken in Produktion. Auf der Content-Seite erkennen 76% der Leser AI-generierten Text binnen drei Sekunden, und das Engagement fällt dann um etwa 47%.

Diese drei Zahlen beschreiben dasselbe Problem aus zwei Blickwinkeln: **AI-Outputs brauchen Guardrails, egal ob Code oder Text.** Heute veröffentlichen wir zwei Claude Code Skill-Packs, die auf diesen Guardrails sitzen.

**Ein Claude Code Skill ist ein verpackter Slash-Command, der deinen AI-Coding-Agenten um eine spezifische Expertise erweitert, lesbar über `SKILL.md` und aufrufbar aus jedem AGENTS.md-kompatiblen Runtime.** Die Skills Wave besteht aus zwei dieser Packs, am selben Tag veröffentlicht, weil die Fehlermuster auf beiden Seiten des AI-Workflows dieselbe Lösung verdienen.

*Ich bin Kacper, AI Engineer bei [Vstorm](https://vstorm.co), einer Applied Agentic AI Engineering Consultancy. Wir haben 30+ produktive AI-Agent-Implementierungen ausgeliefert und öffnen unsere Tools unter [github.com/vstorm-co](https://github.com/vstorm-co).*

## production-stack-skills: 10 Claude Code Skills für produktionsreifen AI-Code

**production-stack-skills ist ein 10-Skill-Pack, das AI-generierten Code in sechs gewichteten Kategorien auditiert und einen 0-bis-100-Score mit priorisiertem Action-Plan zurückgibt.**

Der Flagship-Command ist `/production check`. Du zeigst auf ein Repo, der Skill liest FastAPI-Routes, Postgres-Migrationen, Dockerfiles und Config, und liefert:

- Headline-Score (z.B. "Production Readiness: 34/100")
- Sechs Kategorie-Scores: security, error handling, observability, deployment, data layer, code quality
- Quick-Wins-Sektion mit Punkt-Deltas
- Action-Plan sortiert nach gewichtetem Impact

In den ersten internen Durchläufen über Vstorms Kunden-Repos bewegt allein die Quick-Wins-Sektion den Score konsequent um etwa 30 Punkte. Das ist die Zahl, die ich einem CTO nennen würde, der fragt "was bringt mir ein Vormittag damit".

Die anderen neun Skills teilen sich nach Oberfläche: `/production review`, `/production planner`, `/production fastapi`, `/production postgres`, `/production docker`, `/production deploy`, `/production monitoring`, `/production security` und `/production error-handling`.

## content-skills: 14 Brand-First-Skills gegen AI-Slop

**content-skills ist ein 14-Skill-Pack mit einem `/brand/`-Verzeichnis im Zentrum, sodass jeder Content-Output nach einem fünfminütigen Brand-Interview deine BRAND.md, VOICE.md, VISUAL.md und Voice-Samples liest, bevor das erste Wort geschrieben wird.**

Setup ist der Einstieg. Lauf `/content setup` einmal. Fünf Fragen zu Firma, Voice und visueller Identität. Der Skill schreibt den `/brand/`-Ordner. Ab diesem Moment öffnet jeder Skill im Pack `/brand/` bei jedem Aufruf.

Audit ist der Ausstieg. Lauf `/content audit` auf einem beliebigen Content-Stück und erhalte einen 0-bis-100-Score für Voice-Konsistenz, Anti-Slop-Marker, visuelle Konsistenz und Brand-Alignment.

Dazwischen sitzen 12 Produktions-Skills: strategy, calendar, blog, twitter, linkedin, reddit, hackernews, presentation, infographic, image, video, repurpose. Jeder liest automatisch aus `/brand/`.

## Eine Installation, Dual-CLI, Uninstall-sicher

Beide Packs installieren sich gleich:

```bash
curl -fsSL https://raw.githubusercontent.com/vstorm-co/production-stack-skills/main/install.sh | bash
curl -fsSL https://raw.githubusercontent.com/vstorm-co/content-skills/main/install.sh | bash
```

Jede Installation spiegelt die Skills in `~/.claude/` (Claude Code) und `~/.agents/` (Codex, Amp, alles AGENTS.md-kompatible). Du wählst den Runtime nicht vorab. Ein für Claude Code geschriebener Skill läuft identisch in Codex.

Uninstall ist bewusst langweilig. `/content-skills uninstall` entfernt die Skills. Dein `/brand/` bleibt.

Beide Repos sind MIT.

## Warum zwei Packs am selben Tag

**In 30+ produktiven AI-Agent-Deployments bei Vstorm clustern die Fehler in zwei Formen: Code, der Demo besteht und beim ersten Produktionsvorfall fällt, sowie Content, der nach AI klingt, weil AI ihn geschrieben hat.**

Ein Pack adressiert das erste. Das andere das zweite. Beide nutzen dieselbe Architektur, weil die Lektion, dass Skills monolithische Agenten schlagen, für beide gilt. Du willst nicht "ein AI, das alles macht". Du willst 24 kleine, komponierbare, auditierbare Slash-Commands.

## Häufig gestellte Fragen

### Was sind Claude Code Skills und wie installiere ich sie?

Claude Code Skills sind verpackte Slash-Commands basierend auf einer `SKILL.md`-Datei, die einen Coding-Agenten um spezifische Expertise erweitern. Ein Skill-Pack installierst du mit einem einzigen `curl`-Command, der Files in `~/.claude/` und `~/.agents/` spiegelt. Nach der Installation sind die Slash-Commands sofort verfügbar.

### Was ist der Unterschied zwischen production-stack-skills und content-skills?

production-stack-skills auditiert und härtet AI-generierten Code über security, error handling, observability, deployment, data und code quality mit einem 0-100-Score. content-skills auditiert und produziert markengerechten Content mittels eines `/brand/`-Verzeichnisses, mit Voice-Consistency- und Anti-Slop-Scores.

### Funktionieren diese Skills mit Codex oder nur Claude Code?

Beides. Jedes Install-Script spiegelt Files in `~/.claude/` und `~/.agents/`, sodass derselbe Skill in Claude Code, Codex, Amp und jedem AGENTS.md-kompatiblen Runtime ohne Modifikation läuft.

### Wie lange dauert `/production check` auf einem echten Repo?

Bei einem typischen FastAPI + Postgres Repo mit einigen tausend Zeilen etwa eine Minute. Die Quick-Wins-Sektion oben ist das, was zuerst umgesetzt wird, meist unter fünf Minuten.

### Wann nutze ich einen Claude Code Skill statt eines vollen Agenten?

Skill, wenn die Aufgabe eine abgegrenzte Expertise ist, die explizit aufgerufen wird. Voller Agent, wenn die Aufgabe offen, mehrstufig und tool-übergreifend planend ist. Skills sind komponierbare Bausteine; Agenten orchestrieren sie.
