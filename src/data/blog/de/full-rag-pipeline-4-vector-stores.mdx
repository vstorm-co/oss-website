## Die Architektur: 5 Schritte, jeder konfigurierbar

Jedes RAG-System macht dasselbe: **parse → chunk → embed → store → search**. Der Unterschied liegt darin, wie viele Entscheidungen du bei jedem Schritt treffen musst.

In unserem Template ist jeder Schritt eine austauschbare Abstraktion:

```
Document Upload
  │
  ├── Parse: PyMuPDF (default) | LlamaParse (130+ formats) | python-docx
  │
  ├── Chunk: recursive (default) | markdown | fixed
  │     └── chunk_size=512, overlap=50 (configurable via env vars)
  │
  ├── Embed: OpenAI | Voyage | Gemini (multimodal) | SentenceTransformers (local)
  │     └── dimensions auto-derived from model name
  │
  ├── Store: Milvus | Qdrant | ChromaDB | pgvector
  │
  └── Search: vector | hybrid (BM25 + vector + RRF) | + reranking (Cohere | CrossEncoder)
```

Du waehlst deinen Stack bei der Projektgenerierung. Das Template verdrahtet alles. Kein Glue-Code.

## 4 Vektordatenbanken, 1 Interface

Die groesste Designentscheidung war, Vektordatenbanken austauschbar zu machen. Wir haben `BaseVectorStore` mit vier Backends implementiert:

```python
class BaseVectorStore(ABC):
    async def insert_document(self, collection_name: str, document: Document) -> None
    async def search(self, collection_name: str, query: str, limit: int = 4) -> list[SearchResult]
    async def delete_document(self, collection_name: str, document_id: str) -> None
    async def get_collection_info(self, collection_name: str) -> CollectionInfo
```

**Milvus** — produktionsreif, laeuft als 3 Docker-Services (etcd + MinIO + Milvus). Am besten fuer Deployments im grossen Massstab. Cosine Similarity mit IVF_FLAT-Indexierung.

**Qdrant** — ein einzelner Docker-Service, gute Balance zwischen Leistung und Einfachheit. Unsere Standardempfehlung fuer die meisten Teams.

**ChromaDB** — Embedded-Modus, kein Docker erforderlich. Perfekt fuer Prototyping und lokale Entwicklung. Einfach `pip install chromadb`.

**pgvector** — nutzt dein bestehendes PostgreSQL. Keine neue Infrastruktur. HNSW-Indexierung. Wenn du bereits Postgres hast, ist das die reibungsloseste Option.

Zwischen ihnen wechseln? Eine Umgebungsvariable:

```bash
# In your .env:
VECTOR_STORE=qdrant    # or: milvus, chromadb, pgvector
```

Das Template kuemmert sich automatisch um Connection Strings, Docker-Services, Schema-Erstellung und Index-Konfiguration.

## Hybride Suche: Warum reine Vektorsuche nicht ausreicht

Reine Vektorsuche funktioniert gut fuer semantische Abfragen ("Dokumente ueber Gebaeudesicherheit"). Sie versagt bei exakten Treffern ("finde Vertrag #2024-0847"), weil Embeddings exakte Zeichenketten nicht bewahren.

Unsere hybride Suche kombiniert beides:

```python
async def retrieve(self, query: str, collection_name: str, limit: int = 5):
    # Step 1: Vector search (semantic)
    raw_results = await self.store.search(collection_name, query, limit=limit * fetch_multiplier)

    # Step 2: BM25 keyword search
    if self._hybrid_enabled:
        bm25_results = await self._bm25_search(query, collection_name, limit * fetch_multiplier)
        if bm25_results:
            raw_results = self._rrf_fuse(raw_results, bm25_results)

    # Step 3: Rerank (optional)
    if should_rerank and self.rerank_service:
        results = await self.rerank_service.rerank(query=query, results=raw_results, top_k=limit * 2)

    return results[:limit]
```

Die Fusion verwendet **Reciprocal Rank Fusion (RRF)** — einen einfachen, aber effektiven Algorithmus, der Rankings aus mehreren Quellen kombiniert:

```python
@staticmethod
def _rrf_fuse(vector_results, bm25_results, k=60):
    scores = {}
    for rank, r in enumerate(vector_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    for rank, r in enumerate(bm25_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    return sorted_by_score(scores)
```

Aktiviere es mit einer Umgebungsvariable: `RAG_HYBRID_SEARCH=true`.

## Reranking: Der Qualitaetsmultiplikator

Das initiale Retrieval wirft ein weites Netz aus. Reranking grenzt ein. Wir unterstuetzen zwei Optionen:

**Cohere Reranker** (API) — der schnellste Weg, die Retrieval-Qualitaet zu verbessern. Sende deine Ergebnisse + Abfrage, erhalte sie neu bewertet von einem Modell, das speziell fuer Relevanz-Ranking trainiert wurde:

```python
response = await self.client.rerank(
    query=query,
    documents=[result.content for result in results],
    model="rerank-v3.5",
    top_n=top_k,
)
```

**CrossEncoder** (lokal) — fuehrt ein SentenceTransformers Cross-Encoder-Modell lokal aus. Keine API-Aufrufe, keine Daten verlassen deine Infrastruktur:

```python
pairs = [[query, result.content] for result in results]
scores = self.model.predict(pairs)  # Runs locally on CPU/GPU
```

Die Pipeline ist: 3x mehr Ergebnisse abrufen als benoetigt, rerankieren, Top-k zurueckgeben. Das verbessert konsistent die Praezision, ohne Embeddings oder Vector Store zu beruehren.

## Dokumentenversionierung: SHA256-Deduplizierung

Das erneute Einlesen eines Dokuments sollte keine Duplikate erzeugen. Unsere Pipeline verwendet Content-Hashing:

```python
async def ingest_file(self, filepath, collection_name, replace=True):
    document = await self.processor.process_file(filepath)

    # Check for existing version by source path or content hash
    existing_id = await self._find_existing_by_source(collection_name, source_path)
    if not existing_id:
        existing_id = await self._find_existing_by_hash(collection_name, document.metadata.content_hash)

    # Replace old chunks with new ones
    if existing_id:
        await self.store.delete_document(collection_name, existing_id)

    await self.store.insert_document(collection_name, document)
```

Google Drive-Synchronisierung? Dieselbe Logik — geaenderte Dateien werden neu eingebettet, unveraenderte uebersprungen.

Dimensionen werden automatisch aus dem Modellnamen abgeleitet — keine manuelle Konfiguration:

```python
EMBEDDING_DIMENSIONS = {
    "text-embedding-3-small": 1536,
    "voyage-3": 1024,
    "gemini-embedding-exp-03-07": 3072,
    "all-MiniLM-L6-v2": 384,

```

Gemini ist das Interessante — es unterstuetzt **multimodale Embeddings**. Text und Bilder im selben Vektorraum. Wir nutzen es zur Extraktion von Bildbeschreibungen aus PDFs.

## Die Agent-Integration

RAG wird zum Agent-Tool — `search_knowledge_base` — verfuegbar in allen 5 AI-Frameworks (Pydantic AI, LangChain, LangGraph, CrewAI, DeepAgents):

```python
async def search_knowledge_base(
    query: str,
    collection: str = "documents",
    collections: list[str] | None = None,  # Multi-collection search
    top_k: int = 5,
) -> str:
    """Search with automatic reranking & hybrid search if enabled."""
```

Ergebnisse enthalten Quellenzuordnung: Dateiname, Seitennummer, Chunk-Nummer und Aehnlichkeits-Score. Der System-Prompt des Agenten weist ihn an, Quellen mit `[1]`, `[2]`-Referenzen zu zitieren.

## Wichtigste Erkenntnisse

- **RAG ist eine Pipeline aus 5 Entscheidungen** (parse, chunk, embed, store, search) — unser Template macht jede ohne Codeaenderungen konfigurierbar
- **Reine Vektorsuche verpasst exakte Treffer** — hybrid (BM25 + vector + RRF) erfasst sowohl semantische als auch Keyword-Abfragen
- **Reranking ist die guenstigste Qualitaetsverbesserung** — 3x ueber-abrufen + rerankieren schlaegt konsistent das Tuning von Embeddings
- **Dokumentenversionierung verhindert doppelte Chunks** — SHA256-Content-Hash + Quellpfad-Tracking
- **Eine Umgebungsvariable schaltet alles um** — `VECTOR_STORE=pgvector`, `RAG_HYBRID_SEARCH=true`, `EMBEDDING_MODEL=voyage-3`
