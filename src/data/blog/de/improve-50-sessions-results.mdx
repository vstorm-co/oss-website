> **TL;DR:** Wir haben die `/improve`-Pipeline von pydantic-deep auf 50 echten Entwicklungssitzungen (~3.200 Nachrichten, 7 Tage) ausgeführt und 180+ strukturierte Insights in 7 Typen extrahiert. AgentLearning-Insights (Build-Befehle, Dateipfade, Tool-Chains) dominierten mit 45%, gefolgt von Pattern (20%) und Failure (15%). Die Pipeline schlug 18 Änderungen an Kontextdateien mit kalibrierten Konfidenzwerten vor, MEMORY.md wurde von leer zu einer strukturierten Wissensbasis mit Benutzerpräferenzen, Build-Befehlen, Projektarchitektur und effektiven Debugging-Mustern.

Montag, das Problem: KI-Agenten wiederholen dieselben Fehler, weil jede Sitzung bei null beginnt.

Mittwoch, die Architektur: eine 4-stufige Pipeline (Discovery, Extraction, Raw Traces, Synthesis), die 7 Typen strukturierter Insights aus rohen Sitzungsdaten extrahiert.

Heute, die Daten.

Wir haben `/improve` auf 50 echten Entwicklungssitzungen von [pydantic-deep](https://github.com/vstorm-co/pydantic-deep) ausgeführt, etwa 3.200 Nachrichten verteilt über 7 Tage aktiver Entwicklung. Kein Cherry-Picking, keine synthetischen Daten. Nur die rohe Ausgabe dessen, was eine Pipeline zur Selbstverbesserung findet, wenn man sie auf echte Arbeit richtet.

## TL;DR

- **180+ Insights aus 50 Sitzungen**, automatisch aus rohen Sitzungstranskripten extrahiert, null manuelle Analyse.
- **AgentLearning dominiert mit 45%**, Build-Befehle, Dateipfade, Tool-Chains. Das alltägliche Wissen, das die häufigsten wiederholten Fehler verhindert.
- **MEMORY.md: von leer zu nützlich**, die Pipeline baute eine strukturierte Wissensbasis auf.
- **Konfidenzwerte sind kalibriert**, SOUL.md-Vorschläge durchschnittlich 0,88, MEMORY.md 0,82, AGENTS.md 0,78.
- **50 Sitzungen reichen**, man braucht keine tausend Sitzungen für verwertbare Ergebnisse.

## Das Setup

Die Sitzungen umfassten drei Arbeitstypen:

- **Bug-Fix-Sitzungen** (~15 Sitzungen, durchschn. 45 Nachrichten)
- **Feature-Implementierungen** (~20 Sitzungen, durchschn. 80 Nachrichten)
- **Refactoring-Sitzungen** (~15 Sitzungen, durchschn. 30 Nachrichten)

Wir haben `ImprovementAnalyzer` mit Standardeinstellungen ausgeführt, letzte 7 Tage, alle Sitzungstypen, Konfidenzschwelle bei 0,7:

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)
report = await analyzer.analyze(days=7)
```

Die Pipeline verarbeitete alle 50 Sitzungen in etwa 4 Minuten. Hier ist, was zurückkam.

## Verteilung der Insight-Typen

Die Aufschlüsselung aller 180+ extrahierten Insights:

- **AgentLearning: 45%**, entdeckte Verhaltensweisen, Dateipfade, Build-Befehle, Tool-Chains
- **Pattern: 20%**, wiederkehrende Sequenzen von Tool-Aufrufen
- **Failure: 15%**, Fehler mit Grundursachen und Lösungen
- **UserFact: 10%**, Fakten über den Benutzer (Sprache, Rolle, Präferenzen)
- **Preference + Context + Decision: 10%**, Stilpräferenzen, Architekturfakten, wichtige Entscheidungen

Dass AgentLearning mit 45% dominiert, ist logisch. Das meiste, was ein Agent "lernen" muss, ist praktisches Wissen über seine Umgebung, welche Befehle funktionieren, wo Dateien liegen, was das Build-System erwartet.

## Beispiele pro Sitzung

### Sitzung 1: Bug-Fix (45 Nachrichten)

Der Agent behob einen Testfehler. Hier ist, was `/improve` extrahierte:

**FailureInsight:**

```yaml
description: "Agent tried `pytest tests/` but actual command is `uv run pytest`"
root_cause: "Build system not documented in AGENTS.md"
resolution: "Discovered uv-based workflow after 3 failed attempts"
severity: medium
```

**AgentLearningInsight:**

```yaml
description: "Tests located in tests/ directory, run with `uv run pytest`"
category: tool_chain
confidence: 0.95
```

**PatternInsight:**

```yaml
description: "read → edit → run tests → fix → run tests"
frequency: 4
```

Dieser FailureInsight ist der entscheidende Fund. Der Agent verschwendete 3 Versuche mit direktem `pytest`, bevor er entdeckte, dass das Projekt `uv` verwendet. Ohne `/improve` würde er das in der nächsten Sitzung wiederholen. Mit `/improve` wird der Build-Befehl in AGENTS.md geschrieben, dauerhaft.

### Sitzung 2: Feature-Implementierung (80 Nachrichten)

Längere Sitzung, vielfältigere Insights:

**UserFactInsight:**

```yaml
description: "User speaks Polish"
category: language
confidence: 1.0
```

**PreferenceInsight:**

```yaml
description: "User prefers type-safe solutions with Pydantic models"
evidence: "User rejected dict-based approach twice in favor of Pydantic BaseModel"
confidence: 0.85
```

**ContextInsight:**

```yaml
description: "Project uses Textual for TUI, Pydantic AI for agents"
category: architecture
confidence: 0.9
```

Der PreferenceInsight ist interessant. Der Agent beobachtete, dass der Benutzer Dict-basierte Ansätze zweimal ablehnte, nicht einmal, was Zufall sein könnte, sondern zweimal in verschiedenen Teilen der Codebasis. Das ist genug Signal, um es als Präferenz zu kodieren.

### Sitzung 3: Refactoring (30 Nachrichten)

Kürzere Sitzung, aber wertvolle Insights:

**DecisionInsight:**

```yaml
description: "Moved from dataclasses to Pydantic models for serialization"
rationale: "Pydantic provides model_dump_json() and validation"
confirmed: true
```

**AgentLearningInsight:**

```yaml
description: "model_dump_json() preferred over json.dumps(asdict())"
category: tool_chain
confidence: 0.85
```

DecisionInsights sind der seltenste Typ, aber wohl der wertvollste. Sie erfassen architektonische Entscheidungen, die in zukünftigen Sitzungen nicht versehentlich rückgängig gemacht werden sollten.

## Das Before/After, das am meisten zählt

Der echte Gewinn zeigt sich bei MEMORY.md.

**Vorher:**

```markdown
(leer)
```

**Nachher:**

```markdown
## User

- User's name is Kacper
- Primary language: Polish, code in English
- Prefers type-safe Pydantic models over raw dicts

## Build & Test

- Run tests: `uv run pytest`
- Type check: `uv run ty check`
- Lint: `uv run ruff check --fix`

## Project

- TUI built with Textual
- Agent framework: Pydantic AI
- Structured output via output_type parameter

## Effective Patterns

- Read file → understand → edit → verify with tests
- When debugging: check error message first, then trace to source
```

Das ist strukturiert, umsetzbar und sofort nützlich. Jede zukünftige Sitzung startet mit diesem Kontext statt mit einem leeren Blatt. Der Agent kennt die Build-Befehle, die Sprache des Benutzers, die Projektarchitektur und die effektiven Debugging-Muster, alles automatisch aus rohen Sitzungstranskripten extrahiert.

## Was die Konfidenzwerte verraten

Nicht alle Insights sind gleich:

- **SOUL.md-Vorschläge (durchschn. 0,88):** Höchste Konfidenz. Persönlichkeits- und Stil-Insights sind entweder über viele Sitzungen konsistent oder erscheinen gar nicht.
- **MEMORY.md-Vorschläge (durchschn. 0,82):** Starke Konfidenz. Fakten über Benutzer und Projekt sind verifizierbar.
- **AGENTS.md-Vorschläge (durchschn. 0,78):** Etwas niedriger. Tool-Konfigurationen erfordern mehr Nuancen.

Die Konfidenzschwelle von 0,7 filtert Rauschen heraus. Darunter kommen spekulative Insights. Darüber kommt, was die Pipeline mit hinreichender Sicherheit erkannt hat.

## Warum AgentLearning dominiert

Der 45%-Anteil von AgentLearning ist kein Zufall. Die meisten Agenten-Fehler sind nicht dramatisch. Sie sind alltäglich:

- Falscher Build-Befehl
- Falscher Dateipfad
- Falscher Import-Stil
- Falscher Test-Framework-Aufruf

Genau das erfasst AgentLearningInsight, praktisches, umgebungsspezifisches Wissen, das für einen Menschen trivial ist, aber für einen Agenten, der bei null beginnt, unsichtbar.

PatternInsight mit 20% ist die zweitgrößte Kategorie, weil Agenten Gewohnheiten entwickeln. Der Zyklus "read → edit → test → fix → test" erschien in 4 von 15 Bug-Fix-Sitzungen. Das ist kein Rauschen, das ist ein Workflow, den der Agent als wirksam für diese Codebasis entdeckt hat.

## Wichtige Erkenntnisse

**1. Agenten akkumulieren schnell praktisches Wissen.** 50 Sitzungen reichten für ein umfassendes MEMORY.md.

**2. Das meiste "Lernen" ist langweilig und nützlich.** Build-Befehle und Dateipfade, die alltäglichen Dinge, die echte Reibung verursachen.

**3. Konfidenzwerte sind gut kalibriert.** Die Pipeline übertreibt nicht.

**4. Das System findet Dinge, die Menschen nicht dokumentieren würden.** "Benutzer hat Dict-Ansatz zweimal abgelehnt", das schreibt kein Entwickler auf, aber es hilft dem Agenten.

**5. 50 Sitzungen genügen.** Der Grenznutzen zusätzlicher Sitzungen sinkt schnell für MEMORY.md, wächst aber für PatternInsight und PreferenceInsight.

## Selbst ausprobieren

```bash
pip install pydantic-deep
```

Nach einigen echten Arbeitssitzungen:

```bash
# Im pydantic-deep TUI
/improve
```

Oder programmatisch:

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)
report = await analyzer.analyze(days=7)

# Review proposed changes
for change in report.proposed_changes:
    print(f"{change.target_file}: {change.description} (confidence: {change.confidence})")
```

Das Review-Modal im TUI erlaubt es, jede vorgeschlagene Änderung zu akzeptieren, abzulehnen oder zu bearbeiten, bevor sie angewendet wird. Keine automatischen Schreibvorgänge ohne Ihre Zustimmung.
