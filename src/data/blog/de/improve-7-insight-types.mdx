> **TL;DR:** Der /improve-Befehl in pydantic-deep führt eine 4-stufige Pipeline aus — Discovery, Extraction, Raw Traces, Synthesis — die vergangene Agent-Sessions analysiert und Änderungen an den Kontextdateien des Agenten vorschlägt. Die Extraktionsphase teilt Sessions in 140K-Token-Fenster mit 5-Nachrichten-Overlap und extrahiert 7 strukturierte Insight-Typen mithilfe von Pydantic AI Agents. Die Synthesephase kombiniert Insights über Sessions hinweg, bewertet die Konfidenz und schlägt nur Änderungen über 0.7 vor. Die gesamte Pipeline ist automatisiert — kein manuelles Log-Lesen.

## Die Pipeline

`/improve` ist kein einzelner LLM-Aufruf. Es ist eine 4-stufige Pipeline, orchestriert von einer `ImprovementAnalyzer`-Klasse:

```python
async def analyze(self, days=7, max_sessions=20, focus=None) -> ImprovementReport:
    # 1. Discover sessions
    session_paths = self._discover_sessions(days)
    session_paths = session_paths[:max_sessions]

    # 2. Extract insights per session
    insights: list[SessionInsights] = []
    for path in session_paths:
        session_insights, chunks = await self._extractor.extract(path)
        insights.append(session_insights)

    # 3. Load current context + raw tool sequences
    current_context = self._load_current_context()
    tool_sequences = self._load_tool_sequences(session_paths)

    # 4. Synthesize
    proposed_changes = await self._synthesizer.synthesize(
        insights, current_context, tool_sequences=tool_sequences
    )
    return ImprovementReport(...)
```

Jede Stufe hat eine spezifische Aufgabe. Gehen wir sie durch.

## Stufe 1: Discovery

Die einfachste Stufe. `/improve` durchsucht `~/.pydantic-deep/sessions/` nach Session-Verzeichnissen der letzten N Tage (Standard: 7). Jede Session ist ein Verzeichnis mit Konversationsprotokollen und Tool-Ausführungstraces.

Sie können das Fenster steuern: `days=7` schaut auf die letzte Woche, `max_sessions=20` begrenzt die Analyse. Der `focus`-Parameter sagt dem Extraktor, worauf er sich konzentrieren soll.

Nichts Ausgefallenes — einfache Dateisystem-Traversierung mit Datumsfilterung.

## Stufe 2: Extraction — Das Herzstück

Hier passiert das Wesentliche. Ein `SessionExtractor` verarbeitet jede Session und extrahiert 7 Typen strukturierter Insights.

### Das Chunking-Problem

Agent-Sessions können riesig sein. Eine 2-stündige Coding-Session überschreitet leicht 500K Token. Das kann man einem LLM nicht in einem Durchgang übergeben.

Die Lösung: Sessions in Fenster von maximal 140.000 Token aufteilen, mit 5-Nachrichten-Overlap zwischen Chunks für Kontextkontinuität:

- **MAX_TOKENS_PER_CHUNK = 140.000** — passt komfortabel in die meisten Modell-Kontextfenster
- **OVERLAP_MESSAGES = 5** — die letzten 5 Nachrichten von Chunk N werden die ersten 5 von Chunk N+1
- **Token-Schätzung** — ~4 Zeichen pro Token-Heuristik (schnell, ausreichend für Chunking)
- **Tool-Output-Kürzung** — 70% Kopf, 30% Ende (bewahrt sowohl Anfang als auch Ende langer Outputs)

Die 70/30-Aufteilung für Tool-Output-Kürzung ist bewusst gewählt. Der Anfang eines Tool-Outputs enthält normalerweise den Befehl und die ersten Ergebnisse. Das Ende hat den Endstatus oder die Fehlermeldung. Die Mitte ist oft repetitiver Output, den man sicher weglassen kann.

### Die 7 Insight-Typen

Jeder Chunk wird von einem Pydantic AI Agent mit Structured Output analysiert — kein Freitext, sondern typisierte Dataclasses:

**1. FailureInsight** — Was schief ging, warum und wie es behoben wurde.

```python
@dataclass
class FailureInsight:
    description: str  # What went wrong
    root_cause: str   # Why it happened
    resolution: str   # How it was fixed
    tool_calls: list[str] = field(default_factory=list)
```

Der wertvollste Insight-Typ. Wenn Ihr Agent Tests nicht ausführen konnte, weil er vergessen hat, ein Virtualenv zu aktivieren, erfasst `FailureInsight` das — und die Synthese kann "immer .venv vor pytest aktivieren" in den Kontext des Agenten schreiben.

**2. PatternInsight** — Wiederkehrende Tool-Call-Sequenzen.

```python
@dataclass
class PatternInsight:
    pattern: str      # e.g., "read → edit → test → commit"
    frequency: int
    context: str
```

Wenn Ihr Agent konsistent `read file → edit → run tests → commit` beim Bugfixing macht, ist das ein Muster, das es wert ist, kodifiziert zu werden.

**3. PreferenceInsight** — Stil-Präferenzen des Benutzers aus Korrekturen.

```python
@dataclass
class PreferenceInsight:
    preference: str
    evidence: str     # User correction or explicit request
```

"Verwende keinen `any`-Typ in TypeScript" — erfasst aus einer Session, in der Sie den Agenten korrigiert haben.

**4. ContextInsight** — Architektonische Fakten über das Projekt.

```python
@dataclass
class ContextInsight:
    fact: str
    confidence: float
```

"Das Projekt verwendet Alembic für Datenbankmigrationen" — entdeckt aus Tool-Traces, die `alembic upgrade head`-Befehle zeigen.

**5. DecisionInsight** — Wichtige Entscheidungen während Sessions.

```python
@dataclass
class DecisionInsight:
    decision: str
    reasoning: str
    confirmed: bool
```

"Wechsel von REST zu WebSocket für Streaming-Antworten" — mit der Begründung. Das `confirmed`-Flag zeigt an, ob der Benutzer explizit zugestimmt hat.

**6. UserFactInsight** — Fakten über Sie, den Entwickler.

```python
@dataclass
class UserFactInsight:
    fact: str
    category: str     # identity, role, language, expertise, preference
    confidence: float
```

Kategorien: identity, role, language, expertise, preference. "Benutzer bevorzugt polnische Kommentare, aber englischen Code" — aus tatsächlichen Interaktionen extrahiert.

**7. AgentLearningInsight** — Dinge, die der Agent über die Umgebung entdeckt hat.

```python
@dataclass
class AgentLearningInsight:
    learning: str
    category: str     # tool_chain, file_location, build_command, environment
    evidence: str
    confidence: float
```

"pytest läuft vom Projekt-Root mit `python -m pytest`" — mit dem tatsächlichen Tool-Trace als Beweis.

## Stufe 3: Raw Tool Traces

Hier weicht `/improve` von einfacheren Ansätzen ab.

Anstatt nur die extrahierten Insights an die Synthese weiterzugeben, laden wir auch rohe `tool_log.jsonl`-Dateien aus jeder Session. Dies folgt dem **Meta-Harness-Prinzip**: Rohe Ausführungstraces enthalten diagnostische Informationen, die verloren gehen, wenn ein LLM sie zusammenfasst.

Denken Sie darüber nach — ein LLM, der Insights extrahiert, könnte notieren "der Test ist fehlgeschlagen." Der rohe Trace zeigt *genau*, welche Assertion fehlschlug, was die erwarteten vs. tatsächlichen Werte waren und welcher Befehl ausgeführt wurde.

## Stufe 4: Synthese

Der Synthesizer nimmt alles — extrahierte Insights aus allen Sessions, aktuelle Kontextdateien und rohe Traces — und erzeugt vorgeschlagene Änderungen.

```python
agent: Agent[None, _SynthesisOutput] = Agent(
    model=self._model,
    output_type=_SynthesisOutput,
    instructions=prompt,
)
result = await agent.run(user_prompt)
```

Wieder Structured Output über Pydantic AI. Jede vorgeschlagene Änderung enthält:

- **target_file** — welche Datei aktualisiert werden soll (SOUL.md, AGENTS.md oder MEMORY.md)
- **change_type** — create (neue Datei), append (Abschnitt hinzufügen), update (Abschnitt ersetzen)
- **confidence** — 0.0 bis 1.0. Nur Änderungen mit **>= 0.7** werden vorgeschlagen
- **source_sessions** — welche Sessions Belege lieferten

Die Konfidenz-Schwelle ist entscheidend. Eine einzelne Session mit `npm run build` bedeutet nicht, dass das immer Ihr Build-Befehl ist. Aber wenn 8 von 10 Sessions ihn verwenden — ist die Konfidenz hoch und die Änderung wird vorgeschlagen.

### Änderungsanwendung

Drei Änderungstypen mit klarer Semantik:

- **create** — Neue Datei (z.B. erstmaliges Generieren von MEMORY.md)
- **append** — Inhalt am Ende einer bestehenden Datei hinzufügen
- **update** — Abschnitt nach Überschrift finden, Inhalt bis zur nächsten gleichrangigen oder übergeordneten Überschrift ersetzen

Änderungen durchlaufen einen Review-Schritt im TUI vor der Anwendung.

## State Persistence

`/improve` verfolgt seine eigene Historie in `.pydantic-deep/improve_state.json`:

- Gesamtzahl der Durchläufe
- Analysierte Sessions
- Akzeptierte vs. abgelehnte Änderungen

Das bedeutet, `/improve` analysiert bereits verarbeitete Sessions nicht erneut und lernt aus Ihren Akzeptieren/Ablehnen-Entscheidungen.

## Warum Structured Output wichtig ist

Jeder Extraktions- und Syntheseschritt verwendet Pydantic AI Agents mit typisierten Output-Schemas. Kein Regex-Parsing. Kein "JSON aus dem Markdown-Codeblock extrahieren." Das Modell gibt typisierte Dataclasses zurück, an der Grenze validiert.

Das ist nicht nur sauberer Code — es ist Zuverlässigkeit. Wenn Sie 20 Sessions mit mehreren Chunks verarbeiten, bricht eine einzige fehlformatierte Extraktion die gesamte Pipeline. Structured Output über Pydantic AI eliminiert diesen Fehlermodus.
