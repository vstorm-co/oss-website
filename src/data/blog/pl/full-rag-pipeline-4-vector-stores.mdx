## Architektura: 5 kroków, każdy konfigurowalny

Każdy system RAG robi to samo: **parse → chunk → embed → store → search**. Różnica polega na tym, ile decyzji musisz podjąć na każdym etapie.

W naszym szablonie każdy krok to wymienna abstrakcja:

```
Document Upload
  │
  ├── Parse: PyMuPDF (default) | LlamaParse (130+ formats) | python-docx
  │
  ├── Chunk: recursive (default) | markdown | fixed
  │     └── chunk_size=512, overlap=50 (configurable via env vars)
  │
  ├── Embed: OpenAI | Voyage | Gemini (multimodal) | SentenceTransformers (local)
  │     └── dimensions auto-derived from model name
  │
  ├── Store: Milvus | Qdrant | ChromaDB | pgvector
  │
  └── Search: vector | hybrid (BM25 + vector + RRF) | + reranking (Cohere | CrossEncoder)
```

Wybierasz swój stos podczas generowania projektu. Szablon podłącza wszystko automatycznie. Żadnego kodu klejącego.

## 4 bazy wektorowe, 1 interfejs

Największą decyzją projektową było uczynienie baz wektorowych wymiennymi. Zaimplementowaliśmy `BaseVectorStore` z czterema backendami:

```python
class BaseVectorStore(ABC):
    async def insert_document(self, collection_name: str, document: Document) -> None
    async def search(self, collection_name: str, query: str, limit: int = 4) -> list[SearchResult]
    async def delete_document(self, collection_name: str, document_id: str) -> None
    async def get_collection_info(self, collection_name: str) -> CollectionInfo
```

**Milvus** — klasa produkcyjna, działa jako 3 serwisy Docker (etcd + MinIO + Milvus). Najlepszy do wdrożeń na dużą skalę. Cosine similarity z indeksowaniem IVF_FLAT.

**Qdrant** — jeden serwis Docker, świetna równowaga między wydajnością a prostotą. Nasze domyślne zalecenie dla większości zespołów.

**ChromaDB** — tryb embedded, zero Dockera. Idealny do prototypowania i lokalnego developmentu. Wystarczy `pip install chromadb`.

**pgvector** — korzysta z istniejącego PostgreSQL. Żadnej nowej infrastruktury. Indeksowanie HNSW. Jeśli już masz Postgresa, to opcja z najmniejszym tarciem.

Przełączanie między nimi? Jedna zmienna środowiskowa:

```bash
# In your .env:
VECTOR_STORE=qdrant    # or: milvus, chromadb, pgvector
```

Szablon automatycznie obsługuje connection stringi, serwisy Docker, tworzenie schematów i konfigurację indeksów.

## Wyszukiwanie hybrydowe: dlaczego samo wektorowe nie wystarczy

Czyste wyszukiwanie wektorowe działa dobrze dla zapytań semantycznych ("dokumenty o bezpieczeństwie budynków"). Zawodzi przy dokładnych dopasowaniach ("znajdź umowę #2024-0847"), ponieważ embeddingi nie zachowują dokładnych ciągów znaków.

Nasze wyszukiwanie hybrydowe łączy oba podejścia:

```python
async def retrieve(self, query: str, collection_name: str, limit: int = 5):
    # Step 1: Vector search (semantic)
    raw_results = await self.store.search(collection_name, query, limit=limit * fetch_multiplier)

    # Step 2: BM25 keyword search
    if self._hybrid_enabled:
        bm25_results = await self._bm25_search(query, collection_name, limit * fetch_multiplier)
        if bm25_results:
            raw_results = self._rrf_fuse(raw_results, bm25_results)

    # Step 3: Rerank (optional)
    if should_rerank and self.rerank_service:
        results = await self.rerank_service.rerank(query=query, results=raw_results, top_k=limit * 2)

    return results[:limit]
```

Fuzja wykorzystuje **Reciprocal Rank Fusion (RRF)** — prosty, ale skuteczny algorytm łączący rankingi z wielu źródeł:

```python
@staticmethod
def _rrf_fuse(vector_results, bm25_results, k=60):
    scores = {}
    for rank, r in enumerate(vector_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    for rank, r in enumerate(bm25_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    return sorted_by_score(scores)
```

Włącz jedną zmienną środowiskową: `RAG_HYBRID_SEARCH=true`.

## Reranking: mnożnik jakości

Początkowe wyszukiwanie zarzuca szeroką sieć. Reranking ją zawęża. Wspieramy dwie opcje:

**Cohere Reranker** (API) — najszybszy sposób na poprawę jakości wyszukiwania. Wyślij wyniki + zapytanie, otrzymaj je ponownie ocenione przez model wytrenowany specjalnie do rankingu trafności:

```python
response = await self.client.rerank(
    query=query,
    documents=[result.content for result in results],
    model="rerank-v3.5",
    top_n=top_k,
)
```

**CrossEncoder** (lokalny) — uruchamia model cross-encoder z SentenceTransformers lokalnie. Żadnych wywołań API, żadne dane nie opuszczają twojej infrastruktury:

```python
pairs = [[query, result.content] for result in results]
scores = self.model.predict(pairs)  # Runs locally on CPU/GPU
```

Pipeline wygląda tak: pobierz 3x więcej wyników niż potrzeba, rerankuj, zwróć top-k. To konsekwentnie poprawia precyzję bez dotykania embeddingów czy bazy wektorowej.

## Wersjonowanie dokumentów: deduplikacja SHA256

Ponowne ingestion dokumentu nie powinno tworzyć duplikatów. Nasz pipeline używa hashowania zawartości:

```python
async def ingest_file(self, filepath, collection_name, replace=True):
    document = await self.processor.process_file(filepath)

    # Check for existing version by source path or content hash
    existing_id = await self._find_existing_by_source(collection_name, source_path)
    if not existing_id:
        existing_id = await self._find_existing_by_hash(collection_name, document.metadata.content_hash)

    # Replace old chunks with new ones
    if existing_id:
        await self.store.delete_document(collection_name, existing_id)

    await self.store.insert_document(collection_name, document)
```

Synchronizacja z Google Drive? Ta sama logika — zmienione pliki są ponownie embeddowane, niezmienione pomijane.

## 4 dostawców embeddingów

| Dostawca | Model | Wymiary | Klucz API? |
|----------|-------|---------|------------|
| OpenAI | text-embedding-3-small | 1536 | Tak |
| Voyage | voyage-3 | 1024 | Tak |
| Gemini | gemini-embedding-exp-03-07 | 3072 | Tak |
| SentenceTransformers | all-MiniLM-L6-v2 | 384 | Nie (lokalnie) |

Wymiary są automatycznie określane na podstawie nazwy modelu — żadnej ręcznej konfiguracji:

```python
EMBEDDING_DIMENSIONS = {
    "text-embedding-3-small": 1536,
    "voyage-3": 1024,
    "gemini-embedding-exp-03-07": 3072,
    "all-MiniLM-L6-v2": 384,
}
```

Gemini jest tu najciekawszy — wspiera **multimodalne embeddingi**. Tekst i obrazy w tej samej przestrzeni wektorowej. Używamy go do ekstrakcji opisów obrazów z plików PDF.

## Integracja z agentem

RAG staje się narzędziem agenta — `search_knowledge_base` — dostępnym we wszystkich 5 frameworkach AI (Pydantic AI, LangChain, LangGraph, CrewAI, DeepAgents):

```python
async def search_knowledge_base(
    query: str,
    collection: str = "documents",
    collections: list[str] | None = None,  # Multi-collection search
    top_k: int = 5,
) -> str:
    """Search with automatic reranking & hybrid search if enabled."""
```

Wyniki zawierają atrybucję źródła: nazwa pliku, numer strony, numer chunka i wynik podobieństwa. System prompt agenta instruuje go, aby cytował źródła referencjami `[1]`, `[2]`.

## Kluczowe wnioski

- **RAG to pipeline 5 decyzji** (parse, chunk, embed, store, search) — nasz szablon sprawia, że każda jest konfigurowalna bez zmian w kodzie
- **Samo wyszukiwanie wektorowe pomija dokładne dopasowania** — hybrydowe (BM25 + vector + RRF) łapie zarówno zapytania semantyczne, jak i słowa kluczowe
- **Reranking to najtańsza poprawa jakości** — 3x nadmiarowe pobieranie + reranking konsekwentnie bije strojenie embeddingów
- **Wersjonowanie dokumentów zapobiega duplikatom chunków** — hash SHA256 zawartości + śledzenie ścieżki źródłowej
- **Jedna zmienna środowiskowa przełącza wszystko** — `VECTOR_STORE=pgvector`, `RAG_HYBRID_SEARCH=true`, `EMBEDDING_MODEL=voyage-3`
