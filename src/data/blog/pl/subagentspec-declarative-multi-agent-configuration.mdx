> **TL;DR:** SubAgentSpec to model Pydantic do deklaratywnej konfiguracji multi-agent. Definiuj nazwe agenta, model, system prompt, narzedzia i schemat wynikow w YAML/JSON — otrzymuj type-safe agenty Pydantic AI automatycznie. Wersja v0.2.0 dodaje wsparcie dla wlasnych agentow (fabryki agentow), sledzenie tokenow na poziomie zadan (TaskHandle.usage, get_total_usage()) oraz poprawna serializacje strukturalnych wynikow (model_dump_json() dla Pydantic, json.dumps(asdict()) dla dataclass). Czesc pydantic-ai-subagents: `pip install pydantic-ai-subagents`.

Budujesz system multi-agent. Agent A robi research. Agent B podsumowuje. Agent C waliduje. Kazdy potrzebuje wlasnego modelu, system promptu, narzedzi i schematu wynikow.

Gdzie zyje ta konfiguracja?

W wiekszosci setupow jest rozsiana po plikach Pythona. Agenty definiujesz w jednym module, narzedzia w innym, prompty w pliku z konstantami, nazwy modeli w zmiennych srodowiskowych. Chcesz zmienic model Agenta B z GPT-4.1 na Claude Sonnet? Znajdz odpowiedni plik, znajdz odpowiednia linie, zmien, wdrozenie.

Teraz pomnoz to przez czteroosobowy zespol. Ktos zmienia system prompt w branchu. Ktos inny podmienia model na stagingu. Nikt nie wie co tak naprawde dziala.

Trafilismy na to wystarczajaco czesto w ponad 30 wdrozeniach produkcyjnych, ze zbudowalismy rozwiazanie: [SubAgentSpec](https://github.com/vstorm-co/pydantic-ai-subagents) — deklaratywny model konfiguracji dla systemow multi-agent zbudowany na [Pydantic AI](https://ai.pydantic.dev).

## Problem: Agenty definiowane w kodzie nie skaluja sie w zespolach

Typowy setup multi-agent w kodzie:

```python
research_agent = Agent(
    "openai:gpt-4.1",
    system_prompt="You are a research assistant...",
    tools=[search_web, read_document],
)

summary_agent = Agent(
    "anthropic:claude-sonnet-4-20250514",
    system_prompt="You summarize research findings...",
    result_type=Summary,
)

validation_agent = Agent(
    "openai:gpt-4.1-mini",
    system_prompt="You validate summaries against source material...",
    result_type=ValidationResult,
)
```

Trzy agenty, trzy wybory modeli, trzy system prompty, wszystko zahardkodowane. Chcesz eksperymentowac z roznymi modelami? Edytuj pliki Pythona. Chcesz, zeby nie-developer dostosowal system prompt? Musi ruszac kod. Chcesz wersjonowac konfiguracje agentow oddzielnie od logiki aplikacji? Powodzenia.

To dziala w prototypie. W zespolach produkcyjnych sie rozpada.

## SubAgentSpec: Deklaratywna konfiguracja, type-safe agenty

SubAgentSpec to model Pydantic reprezentujacy kompletna specyfikacje agenta. Definiujesz go w YAML lub JSON, a `pydantic-ai-subagents` tworzy z niego w pelni skonfigurowane agenty Pydantic AI.

```yaml
# agents.yaml
agents:
  - name: research
    model: openai:gpt-4.1
    system_prompt: |
      You are a research assistant. Search for information
      and return structured findings.
    tools:
      - search_web
      - read_document
    
  - name: summarizer
    model: anthropic:claude-sonnet-4-20250514
    system_prompt: |
      You summarize research findings into concise,
      actionable summaries.
    result_type: Summary
    
  - name: validator
    model: openai:gpt-4.1-mini
    system_prompt: |
      You validate summaries against source material.
      Flag any unsupported claims.
    result_type: ValidationResult
```

Zaladuj w Pythonie:

```python
from pydantic_ai_subagents import SubAgentSpec, create_agent_factory_toolset

# Load specs from YAML
specs = SubAgentSpec.from_yaml("agents.yaml")

# Create agent factory toolset
toolset = create_agent_factory_toolset(specs)
```

Konfiguracja agentow jest teraz oddzielona od logiki aplikacji. Przechowuj ja w version control, podmieniaj miedzy srodowiskami, pozwol zespolowi iterowac na promptach bez ruszania Pythona.

Specyfikacje to modele Pydantic — dostajesz walidacje, serializacje i autouzupelnianie w IDE za darmo. Literowka w nazwie modelu? Blad walidacji przy ladowaniu, nie crash runtime 20 minut w sesji.

## Wsparcie dla wlasnych agentow: wzorzec `agent_factory`

Nie kazdy subagent to prosty `Agent()`. Moze twoj agent do researchu potrzebuje wlasnych dependencji. Moze uzywa specjalistycznego frameworka. Moze potrzebuje middleware, pamieci lub specyficznej logiki inicjalizacji.

SubAgentSpec v0.2.0 wprowadzil pola `agent` i `agent_factory`:

```python
from pydantic_ai import Agent
from pydantic_ai_subagents import create_agent_factory_toolset

# Opcja 1: Przekaz gotowego agenta
custom_research_agent = Agent(
    "openai:gpt-4.1",
    system_prompt="Custom research agent with special deps",
    deps_type=ResearchDeps,
)

toolset = create_agent_factory_toolset(
    specs,
    agents={"research": custom_research_agent},
)

# Opcja 2: Uzyj funkcji fabryki
def create_research_agent(spec: SubAgentSpec) -> Agent:
    agent = Agent(
        spec.model,
        system_prompt=spec.system_prompt,
        deps_type=ResearchDeps,
    )
    agent.tool(custom_search)
    return agent

toolset = create_agent_factory_toolset(
    specs,
    agent_factories={"research": create_research_agent},
)

# Opcja 3: Ustaw domyslna fabryke dla WSZYSTKICH agentow
toolset = create_agent_factory_toolset(
    specs,
    default_agent_factory=my_custom_factory,
)
```

Lancuch priorytetow: `agent` (gotowy) > `agent_factory` (fabryka per-spec) > `default_agent_factory` (globalna domyslna) > prosty `Agent()`.

YAML definiuje *co* kazdy agent powinien byc. Fabryka definiuje *jak* go zbudowac.

## Sledzenie tokenow miedzy subagentami

Kiedy uruchamiasz wielu subagentow, musisz wiedziec dokad ida twoje tokeny. Ktory agent jest drogi? Ktory efektywny?

v0.2.0 dodal sledzenie zuzycia tokenow na poziomie zadan:

```python
from pydantic_ai_subagents import check_task, get_total_usage

# Po uruchomieniu zadan...
result = check_task(task_handle)
# Wyswietla: input_tokens, output_tokens, total_tokens

# Agregacja miedzy wszystkimi zadaniami
total = get_total_usage(all_task_handles)
# Zwraca: RunUsage z kumulatywnymi licznikami tokenow
```

Kazdy `TaskHandle` przechowuje obiekt `RunUsage` z uruchomienia subagenta. `check_task` wyswietla liczniki tokenow input/output dla pojedynczego zadania. `get_total_usage()` agreguje miedzy wszystkimi uchwytami zadan.

Sledzenie na poziomie zadan mowi ci, ktore *logiczne zadanie* zuzylo jakie zasoby, nie tylko ktore wywolanie modelu.

## Serializacja strukturalnych wynikow

Gdy subagenty zwracaja strukturalne wyniki, serializacja ma znaczenie. SubAgentSpec obsluguje zarowno modele Pydantic jak i dataclassy:

```python
# Dla modeli Pydantic → model_dump_json()
class Summary(BaseModel):
    title: str
    key_points: list[str]
    confidence: float

# Dla dataclass → json.dumps(asdict())
@dataclass
class ValidationResult:
    is_valid: bool
    issues: list[str]
```

Serializacja zachowuje poprawna strukture JSON dla agenta nadrzednego. Modele Pydantic uzywaja `model_dump_json()` do wydajnej, schema-aware serializacji. Dataclassy uzywaja `json.dumps(asdict())` dla kompatybilnosci JSON bez wymagania Pydantic.

To ma znaczenie przy laczeniu agentow: wynik sumaryzatora staje sie wejsciem walidatora. Jesli serializacja traci strukture (np. konwersja przez `str()`), agent odbierajacy musi parsowac wolny tekst zamiast danych strukturalnych.

## Wszystko razem

Pelny pipeline multi-agent z SubAgentSpec:

```python
from pydantic_ai_subagents import (
    SubAgentSpec,
    create_agent_factory_toolset,
    check_task,
    get_total_usage,
)

# 1. Load agent specs from YAML
specs = SubAgentSpec.from_yaml("agents.yaml")

# 2. Create toolset with custom factories where needed
toolset = create_agent_factory_toolset(
    specs,
    agent_factories={"research": create_research_agent},
    default_agent_factory=standard_factory,
)

# 3. Run tasks
research_handle = await toolset.run("research", query="...")
summary_handle = await toolset.run("summarizer", input=research_handle.result)
validation_handle = await toolset.run("validator", input=summary_handle.result)

# 4. Check token usage per task
for handle in [research_handle, summary_handle, validation_handle]:
    check_task(handle)

# 5. Get total usage
total = get_total_usage([research_handle, summary_handle, validation_handle])
print(f"Total: {total.total_tokens} tokens")
```

Konfiguracja zyje w YAML. Logika biznesowa zyje w Pythonie. Sledzenie tokenow jest automatyczne. Strukturalne wyniki przeplywaja miedzy agentami bez hackow serializacji.

## Kluczowe wnioski

- **SubAgentSpec** oddziela konfiguracje agentow (YAML/JSON) od logiki aplikacji (Python) — bezpieczniejsza iteracja, lepsza wspolpraca zespolowa
- **Fabryki wlasnych agentow** pozwalaja mieszac proste i zlozone agenty w jednym systemie
- **Sledzenie tokenow na poziomie zadan** pokazuje, ktore logiczne zadanie zuzywa zasoby
- **Serializacja strukturalnych wynikow** zachowuje strukture JSON miedzy lancuchami agentow
- **Walidacja Pydantic** lapie bledy konfiguracji przy ladowaniu, nie w runtime
