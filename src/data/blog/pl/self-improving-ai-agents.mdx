> **TL;DR:** Agenty AI powtarzaja te same bledy miedzy sesjami, bo kazda konwersacja zaczyna sie od zera. Komenda `/improve` w pydantic-deep analizuje przeszle sesje, wyciaga 7 typow strukturalnych wnioskow (AgentLearning, Failure, Pattern, UserFact, Preference, Context, Decision) i automatycznie proponuje aktualizacje plikow kontekstowych (SOUL.md, AGENTS.md, MEMORY.md). Bez fine-tuningu, bez RAG — tylko strukturalna ekstrakcja z tego, czego agent juz doswiadczyl, wpleciona z powrotem w kontekst. Kazdy powtorzony blad kosztuje kredyty API i czas inzyniera. /improve zamienia porazki w trwale usprawnienia.

Uruchamiasz swojego agenta AI w poniedzialek. Szuka plikow konfiguracyjnych w glownym katalogu projektu zamiast w katalogu `config/`. Poprawiasz go. Dziala.

Uruchamiasz tego samego agenta we wtorek. Znow szuka konfiguracji w glownym katalogu. Znow poprawiasz. Znow dziala.

Uruchamiasz go w srode. Ten sam blad. Ta sama poprawka. Te same zmarnowane tokeny i czas.

To nie jest problem inteligencji modelu. GPT-4.1, Claude, Gemini — wszystkie to robia. Kazda sesja zaczyna sie od czystej karty. Okno kontekstowe sie zapelnia, sesja sie konczy i wszystko, czego agent sie nauczyl, znika.

Po wdrozeniu ponad 30 produkcyjnych agentow AI w [Vstorm](https://vstorm.co) przestalismy akceptowac to jako normalne. Wzorzec byl stalym: agenty powtarzaja te same tryby awarii miedzy sesjami, bo nie maja mechanizmu uczenia sie z przeszlych doswiadczen.

Wiec zbudowalismy `/improve`.

## Problem: bezstanowe agenty w stanowym swiecie

Okreslmy precyzyjnie, co oznacza "uczenie sie agenta". Nie mowie o fine-tuningu, RLHF ani petlach treningowych. Mowie o czyms prostszym i bardziej praktycznym: **agencie, ktory nie powtarza tych samych bledow**.

Dzisiejsze frameworki agentowe — [Pydantic AI](https://ai.pydantic.dev/), LangChain, CrewAI, AutoGen — traktuja sesje jako niezalezne zdarzenia. Agent ma system prompt, moze jakies pliki kontekstowe i historie konwersacji, ktora resetuje sie co sesje. Jesli agent odkryje, ze Twoje testy sa w `tests/` (nie `test/`), ze komenda budowania to `make build` (nie `npm run build`), albo ze preferujesz taby zamiast spacji — ta wiedza umiera wraz z sesja.

Obejscie, ktore stosuje wiekszosc zespolow: reczna aktualizacja system promptu lub plikow kontekstowych. Po obserwowaniu jak agent sie meczy, developer otwiera `CLAUDE.md` lub odpowiednik i dodaje notatke. Dziala. Ale sie nie skaluje.

Rozwazmyz, co dzieje sie przez 50 sesji:

- **Sesja 12:** Agent odkrywa, ze `docker-compose.yml` jest w `infra/`, nie w glownym katalogu
- **Sesja 23:** Agent uczy sie, ze narzedzie do migracji to Alembic, nie surowy SQL
- **Sesja 31:** Agent ustala, ze pipeline CI uruchamia `pytest -x --tb=short`
- **Sesja 47:** Agent odkrywa, ze monorepo uzywa zaleznosci na poziomie workspace

Kazde z tych odkryc kosztuje tokeny, czas i cierpliwosc czlowieka. I wszystkie to rzeczy, ktore agent juz kiedys "wiedzial" — tylko w innej sesji, ktora juz nie istnieje.

## Przedstawiamy /improve: automatyczna analiza sesji

`/improve` to komenda w [pydantic-deep](https://github.com/vstorm-co/pydantic-deep), naszym open-source'owym frameworku agentow AI, ktory implementuje "deep agent pattern" (ta sama architektura co Claude Code, Manus i Devin).

Na wysokim poziomie `/improve` robi trzy rzeczy:

1. **Analizuje przeszle sesje** — czyta logi konwersacji z ostatnich N dni
2. **Wyciaga strukturalne wnioski** — identyfikuje 7 typow wnioskow z tych sesji
3. **Aktualizuje pliki kontekstowe** — proponuje zmiany w SOUL.md, AGENTS.md i MEMORY.md

Rezultat: agent staje sie madrzejszy po kazdej partii sesji, bez recznej edycji plikow.

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)
report = await analyzer.analyze(days=7)
# report.proposed_changes → automatic MEMORY.md updates
```

Jedna komenda. Siedem dni sesji przeanalizowanych. Pliki kontekstowe zaktualizowane.

## 7 typow wnioskow

Sednem `/improve` jest system ekstrakcji wnioskow. Zamiast wrzucac surowe dane sesji do promptu, kategoryzuje odkrycia na 7 strukturalnych typow:

**1. AgentLearning** — rzeczy, ktore agent odkryl metoda prob i bledow:

```python
@dataclass
class AgentLearningInsight:
    learning: str     # e.g. "tests are in tests/ dir"
    category: str     # tool_chain, file_location, build_command...
    evidence: str     # Tool calls that proved this
    confidence: float # 0.0-1.0
```

*Przyklad: "Projekt uzywa `make test`, nie `pytest` bezposrednio. Pewnosc: 0.95 (zaobserwowano w 4 sesjach)."*

**2. Failure** — co poszlo nie tak i jak zostalo rozwiazane:

```python
@dataclass
class FailureInsight:
    description: str  # What went wrong
    root_cause: str   # Why it happened
    resolution: str   # How it was fixed
```

*Przyklad: "Agent probowal edytowac `/src/main.py`, ale faktyczna sciezka to `/app/main.py`. Przyczyna: zalozone standardowa strukture projektu Python. Rozwiazanie: zawsze sprawdzaj drzewo plikow najpierw."*

**3. Pattern** — powtarzajace sie workflow'y agenta:

```python
@dataclass
class PatternInsight:
    pattern: str      # e.g. "read -> edit -> test -> commit"
    frequency: int    # How many times observed
    context: str      # When this pattern applies
```

*Przyklad: "Czytaj plik -> edytuj -> uruchom testy -> commituj. Czestotliwosc: 12. Kontekst: wszystkie zadania modyfikacji kodu."*

**4. UserFact** — rzeczy, ktore agent nauczyl sie o preferencjach i srodowisku uzytkownika.

**5. Preference** — jak uzytkownik lubi, zeby rzeczy byly robione (styl kodu, styl komunikacji, wybor narzedzi).

**6. Context** — fakty specyficzne dla projektu (stos technologiczny, decyzje architektoniczne, konwencje zespolu).

**7. Decision** — kluczowe decyzje podjete podczas sesji, ktore powinny byc zachowane.

Kazdy typ wniosku ma konkretna strukture, konkretne wymagania dowodowe i konkretny plik docelowy (SOUL.md, AGENTS.md lub MEMORY.md).

## Dlaczego to wazne: koszt powtorzen

Uzasadnienie biznesowe dla samouczacych sie agentow jest proste:

**Zmarnowane kredyty API.** Kazdy powtorzony blad spala tokeny. Agent, ktory spedza 5 wywolan narzedzi na odkrywaniu struktury plikow — w kazdej sesji — moze zuzywac 2 000-3 000 dodatkowych tokenow na sesje. Przez 50 sesji tygodniowo to 100K-150K zmarnowanych tokenow. Przy cenach GPT-4.1 czy Claude to sie sumuje.

**Zmarnowany czas inzynierow.** Za kazdym razem, gdy inzynier obserwuje znany blad agenta i recznie go poprawia, to narzut na przelaczanie kontekstu. Pomnoz przez liczbe developerow w zespole uzywajacych agentow AI codziennie.

**Kumulujaca sie wiedza.** Odwrotnoscia kumulujacych sie bledow jest kumulujaca sie wiedza. Agent, ktory uczy sie z 50 sesji, ma kontekst, ktorego reczne udokumentowanie zajaloby czlowiekowi godziny. I w przeciwienstwie do recznej dokumentacji, `/improve` wylapuje rzeczy, o ktorych ludzie nie pomysleliby, zeby je zapisac — dziwactwa lancuchow narzedzi, wzorce bledow, niejawne preferencje.

## Czym /improve NIE jest

Zeby jasno okreslic granice:

- **To nie fine-tuning.** Wagi modelu sie nie zmieniaja. /improve aktualizuje pliki kontekstowe ladowane do system promptu.
- **To nie RAG.** Nie ma bazy wektorowej ani wyszukiwania semantycznego. Wnioski sa strukturalne, skategoryzowane i zapisane do konkretnych plikow.
- **To nie magia.** Jesli Twoj agent fundamentalnie nie potrafi wykonac zadania, /improve nie pomoze. Optymalizuje to, co agent juz umie robic.

Pomysl o tym jak o roznicy miedzy developerem, ktory prowadzi osobista wiki z wyciagnietymi wnioskami, a takim, ktory codziennie zaczyna od zera. Te same umiejetnosci, diametralnie rozna efektywnosc.

## Jak zaczac

pydantic-deep jest open source:

```bash
pip install pydantic-deep
```

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer
from pydantic_deep.config import get_sessions_dir
from pathlib import Path

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)

# Analyze last 7 days of sessions
report = await analyzer.analyze(days=7)

# Review proposed changes
for change in report.proposed_changes:
    print(f"File: {change.target_file}")
    print(f"Action: {change.action}")
    print(f"Content: {change.content}")
```

## Kluczowe wnioski

- **Agenty AI sa domyslnie bezstanowe.** Kazda sesja zaczyna sie od zera, nawet jesli agent rozwiazal ten sam problem wczoraj.
- **Samoudoskonalanie nie wymaga fine-tuningu.** Strukturalna ekstrakcja wnioskow z przeszlych sesji, wpleciona z powrotem w pliki kontekstowe, wystarczy do znaczacej poprawy.
- **7 typow wnioskow pokrywa pelne spektrum uczenia sie:** porazki, wzorce, wnioski, fakty o uzytkowniku, preferencje, kontekst i decyzje.
- **Koszt powtorzen sie kumuluje.** Zmarnowane tokeny + zmarnowany czas inzynierow, kazda sesje, kazdy dzien.
- **/improve automatyzuje to, co dobrzy inzynierowie robia recznie:** przeglad przeszlej pracy i zapisywanie wyciagnietych wnioskow.
