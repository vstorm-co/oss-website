## 1. Problem: agenty AI są domyślnie bezstanowe

Za każdym razem gdy zaczynasz sesję agenta, agent zaczyna od zera.

Nie pamięta, że we wtorek szukał na GitHubie "pydantic" i dostał 0 wyników, potem "pydantic-ai" i dostał 47. Nie pamięta, że preferujesz async. Nie pamięta niczego.

/improve to nasze rozwiązanie tego problemu.

## 2. Co robi /improve: pipeline

/improve to pipeline post-sesyjny, który wyodrębnia co wydarzyło się w sesji, destyluje wnioski i zapisuje je do trwałego pliku wiedzy (MEMORY.md), który przyszłe sesje ładują automatycznie.

Przepływ:

```
Sesja kończy się
    ↓
tool_log.jsonl zapisany (surowe wywołania narzędzi, do 20K znaków)
    ↓
Komenda /improve
    ↓
Agent ekstrakcji czyta surowe ślady narzędzi
    ↓
Wyodrębnia: UserFactInsight + AgentLearningInsight
    ↓
Agent syntezy: wnioski + surowe ślady → ustrukturyzowane ulepszenia
    ↓
MEMORY.md zaktualizowany
    ↓
Następna sesja: agent ładuje MEMORY.md przy starcie
```

## 3. UserFactInsight vs AgentLearningInsight

**UserFactInsight**, fakty o _Tobie_:

- "Użytkownik preferuje asynchroniczne wzorce Pythona"
- "Użytkownik pracuje nad pydantic-deep"
- "Projekty użytkownika są w ~/Projects/"

**AgentLearningInsight**, fakty o _domenie, narzędziach, wzorcach_:

- "GitHub: lepiej użyć 'pydantic-ai' niż 'pydantic'"
- "Narzędzie X kończy się cichym błędem przy pustym wejściu"

UserFactInsight personalizuje zachowanie. AgentLearningInsight poprawia poprawność.

## 4. Odkrycie Meta-Harness: surowe ślady biją podsumowania

Testowaliśmy dwa podejścia dla danych wejściowych agenta syntezy. Surowe ślady narzędzi wygrały wyraźnie.

Dlaczego:

**Podsumowanie:** "Agent przeszukał GitHub i znalazł odpowiednie repozytorium."

**Surowy ślad:**

```python
browser.navigate("github.com/search?q=pydantic+agent")
# → 0 wyników

browser.navigate("github.com/search?q=pydantic-ai+agent")
# → 47 wyników

browser.click("vstorm-co/pydantic-deep")
# → sukces
```

Z surowego śladu agent syntezy może wyodrębnić: "Używaj 'pydantic-ai' a nie 'pydantic' do wyszukiwań na GitHub."

Z podsumowania: nic użytecznego.

**Podsumowania optymalizują pod czytelność, ale synteza potrzebuje ścieżki decyzji.** Surowy ślad to rzeczywisty dowód.

Praktyczny wniosek: `tool_log.jsonl` przechowuje teraz do 20K znaków surowych danych wywołań narzędzi per sesja (poprzedni limit: 2K, który cicho obcinał większość rzeczywistych sesji).

## 5. MEMORY.md jako ewoluująca baza wiedzy agenta

MEMORY.md znajduje się w `.pydantic-deep/main/MEMORY.md`. Agent ładuje go przy starcie przed przetworzeniem jakiegokolwiek promptu użytkownika.

Po kilku sesjach z /improve może wyglądać tak:

```markdown
# Memory

## User Facts

- Prefers async Python patterns
- Working on pydantic-deep (github.com/vstorm-co/pydantic-deep)
- Projects live in ~/Projects/

## Agent Learnings

- GitHub: use "pydantic-ai" not "pydantic" for better search results
- Check for empty input before calling file read tools
```

Ten plik rośnie i udoskonala się z każdym uruchomieniem /improve.

## 6. Naprawione błędy, rzeczywistość wdrożeń

- **Crash przy `{}` w szablonach promptów**, `str.format()` rzucał KeyError; naprawione przez escape'owanie
- **Ciche błędy**, teraz raportuje `failed_sessions` + `last_error`
- **Brak klucza API**, teraz ładuje klucze z keystora przed tworzeniem agentów
- **Zły model**, hardcoded OpenRouter fallback zastąpiony modelem z `config.toml`
- **Niezgodność ścieżki MEMORY.md**, pisał do `./MEMORY.md` zamiast `.pydantic-deep/main/MEMORY.md`

## Kluczowe wnioski

- Agenty AI są domyślnie bezstanowe, /improve to naprawia
- Dwa typy wniosków: UserFactInsight (o Tobie) i AgentLearningInsight (o domenie)
- Surowe ślady biją podsumowania, ścieżka decyzji to rzeczywisty sygnał uczenia
- MEMORY.md jako ewoluująca baza wiedzy agenta
- 5 naprawionych błędów w v0.3.4, niezawodność jest trudna
- Brak retrainingu, ulepszenie pochodzi z kontekstu, nie z wag modelu
