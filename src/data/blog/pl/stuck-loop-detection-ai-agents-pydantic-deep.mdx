> **TL;DR:** Większość pętli agentów to nie awarie modelu — to mechaniczne powtórzenia, których sam model nie rozpoznaje. pydantic-deep v0.3.8 wprowadza StuckLoopDetection, capability wychwytującą trzy wzorce pętli (powtarzające się identyczne wywołania, naprzemienna A-B-A-B, pętle no-op) zanim zmarnują tokeny. Konfigurowalny próg, konfigurowalna akcja (ModelRetry lub StuckLoopError), izolacja stanu per-run dla równoległych przebiegów. Domyślnie włączone.

W niedzielę opublikowałem przegląd pięciu wydań, które wysłaliśmy w ciągu dwóch tygodni. Dziś zaczynam szczegółowe analizy. Na początek: StuckLoopDetection — capability wydana w v0.3.8, którą powinniśmy byli zbudować miesiące temu.

Oto incydent, który to uwidocznił.

Agent do kodowania pracował nad zadaniem refaktoryzacji przez noc. Zadanie: przestrukturyzowanie importów modułu i aktualizacja zależnych plików. Agent zaszedł gdzieś do połowy, natrafił na plik z nietypo­wym wzorcem importów, próbował go odczytać, nie mógł poprawnie sparsować wyniku i domyślił się, że powinien przeczytać plik ponownie.

Rano: 47 wywołań `read_file` na tej samej ścieżce. $12 kosztów API. Zero postępu w rzeczywistym zadaniu.

Model nie był "zepsuty". Każde wywołanie wyglądało lokalnie uzasadnione — agent przeczytał plik, zdezorientował się, spróbował ponownie. Z perspektywy modelu: starał się. Z zewnątrz: utknął.

## Dlaczego prompt nie wystarczy

Oczywistą odpowiedzią na pętle agentów są lepsze system prompty. "Nie powtarzaj wywołań narzędzi." "Jeśli dostaniesz ten sam wynik dwa razy, spróbuj innego podejścia."

To działa czasem. Problemy:

1. **Model często nie rozpoznaje pętli jako pętli.** Każde powtórzone wywołanie wydaje się lokalnie uzasadnione. Agent nie "ignoruje" instrukcji — naprawdę nie widzi wzorca między wywołaniami.
2. **Zgodność z promptem pogarsza się pod obciążeniem poznawczym.** Długotrwałe zadania z wieloma wywołaniami narzędzi, złożonymi kontekstami i wieloma ograniczeniami wypychają model w kierunku ignorowania instrukcji dla przypadków granicznych.
3. **Musisz dodać instrukcję do każdego agenta osobno.** Nie ma standardowego miejsca na logikę "zapobiegania pętlom" stosującą się wszędzie.

Detekcja na poziomie capability rozwiązuje wszystkie trzy. Nie prosi modelu o monitorowanie siebie — monitoruje z zewnątrz.

## Trzy wzorce pętli

Po przejrzeniu logów sesji produkcyjnych odkryliśmy, że pętle agentów dzielą się na trzy odrębne kategorie.

### Wzorzec 1: Powtarzające się identyczne wywołania

Najprostszy wzorzec. Agent wywołuje to samo narzędzie z tymi samymi argumentami N razy z rzędu.

```
Tura 12: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
Tura 13: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
Tura 14: read_file(path="src/config.json")  → {"imports": [...], "unknown_field": ...}
```

**Dlaczego to się dzieje:** Agent nie może przetworzyć wyniku i nie ma innego narzędzia, do którego mógłby się odwołać. Domyślne zachowanie: spróbować tego samego ponownie, licząc na inny wynik.

**Domyślny próg:** 3 identyczne wywołania. Po trzecim wyzwala się interwencja.

### Wzorzec 2: Naprzemienna pętla A-B-A-B

Bardziej subtelna. Agent naprzemiennie używa dwóch narzędzi, każde podając niekompletny wynik drugiemu.

```
Tura 8:  list_directory(path="src/")
Tura 9:  read_file(path="src/main.py")
Tura 10: list_directory(path="src/")
Tura 11: read_file(path="src/main.py")
Tura 12: list_directory(path="src/")
```

**Dlaczego to się dzieje:** Narzędzie A zwraca coś, co sugeruje Narzędzie B, Narzędzie B zwraca coś, co sugeruje Narzędzie A, i agent cykluje w nieskończoność. Żadne narzędzie nie daje agentowi dość, żeby posunąć się do przodu, ale oba dają dość, żeby uzasadnić kolejne wywołanie.

Ten wzorzec jest szczególnie kosztowny, bo wygląda jak postęp — agent wywołuje różne narzędzia, zmienia kontekst — ale nigdzie nie dociera.

### Wzorzec 3: Pętle no-op

Agent wykonuje to samo wywołanie i dostaje ten sam wynik, ale kontynuuje mimo to.

```
Tura 5:  write_file(path="output.txt", content="result: error")  → success
Tura 6:  write_file(path="output.txt", content="result: error")  → success
Tura 7:  write_file(path="output.txt", content="result: error")  → success
```

**Dlaczego to się dzieje:** Agent próbuje "zatwierdzić" wynik, ale nie zdaje sobie sprawy, że zapis już się dokonał. Lub czeka na jakiś efekt dalszy, który nigdy nie nadchodzi.

## Implementacja

StuckLoopDetection to capability — podpina się do cyklu życia wywołań narzędzi agenta.

```python
from pydantic_deep import create_deep_agent
from pydantic_deep.capabilities import StuckLoopDetection

# Prosto: użyj domyślnego
agent = create_deep_agent(
    model="anthropic:claude-opus-4-6",
    stuck_loop_detection=True,  # domyślnie: True, próg: 3
)

# Niestandardowy: skonfiguruj próg i akcję
agent = create_deep_agent(
    model="anthropic:claude-opus-4-6",
    capabilities=[
        StuckLoopDetection(
            max_repeated=5,    # dopuść do 5 identycznych wywołań przed interwencją
            action="warn",     # "warn" = ModelRetry, "error" = StuckLoopError
        )
    ],
)
```

### Co się dzieje po wykryciu pętli

**`action="warn"` (domyślnie):** Capability wyzwala `ModelRetry`. Model otrzymuje komunikat:

```
You have called read_file(path="src/config.json") 3 times with identical arguments
and received the same result. This indicates a stuck loop. You must try a different
approach — use a different tool, request help, or report that the task cannot be
completed with available tools.
```

Przez większość czasu to wystarczy. Model przestawia się na inne narzędzie lub eskaluje.

**`action="error"`:** Podnosi `StuckLoopError`. Wykonanie zatrzymuje się. Użyteczne dla automatycznych pipeline'ów.

```python
from pydantic_deep.capabilities import StuckLoopDetection, StuckLoopError

try:
    result = await agent.run("refactor the imports in src/")
except StuckLoopError as e:
    print(f"Agent utknął: wykryto wzorzec {e.pattern}")
    # obsłuż elegancko — eskaluj, spróbuj ponownie z innym modelem, itp.
```

### Izolacja stanu per-run

Jeśli uruchamiasz `agent.run()` współbieżnie dla wielu zadań, stan detekcji pętli nie może wyciekać między przebiegami. Run 1 wywołujący `read_file` trzy razy nie powinien wyzwalać detekcji dla Run 2.

```python
async def run_task(task: str):
    # Każde wywołanie agent.run() dostaje izolowany stan detekcji pętli
    # poprzez for_run() — współdzielona instancja agenta, izolowane capabilities per-run
    return await agent.run(task)

# Bezpiecznie uruchamiać współbieżnie
results = await asyncio.gather(
    run_task("analyze src/module_a.py"),
    run_task("analyze src/module_b.py"),
)
```

Jest to obsługiwane automatycznie. Nie potrzebujesz tworzyć osobnych instancji agenta dla równoległych przebiegów.

## Uzasadnienie biznesowe

47-wywołaniowa pętla przy cenach Claude Opus kosztuje około $12 za zadanie, które powinno kosztować $0.50. W skali — wielu agentów, wiele współbieżnych zadań, nocne przebiegi — to się kumuluje szybko.

Rachunek dla dodania `stuck_loop_detection=True`:

- **Koszt:** zero (domyślnie włączone, żadnych dodatkowych wywołań API)
- **Opóźnienie:** pomijalny (detekcja in-memory, żadnych wywołań LLM)
- **Ryzyko:** niskie (tryb warn wyzwala ModelRetry, nie twarde zatrzymanie)
- **Korzyść:** wychwytuje ogon runaway loops zanim wyczerpią budżety

## Kluczowe wnioski

- **Pętle agentów są zazwyczaj mechaniczne, nie awariami modelu.** Model nie rozpoznaje wzorca; detekcja musi odbywać się poza modelem.
- **Trzy wzorce:** powtarzające się identyczne wywołania, naprzemienna A-B-A-B, pętle no-op — każda ma inną przyczynę źródłową.
- **Detekcja na poziomie capability** oznacza, że stosuje się jednolicie bez prompt engineeringu per-agent.
- **Izolacja per-run** (`for_run()`) sprawia, że jest bezpieczna dla równoległych wywołań `agent.run()`.
- **`action="warn"`** jest właściwą wartością domyślną — ModelRetry przed StuckLoopError, elegancko przed twardym zatrzymaniem.
- **Domyślnie włączone** w `create_deep_agent()` — nie musisz o tym myśleć.
