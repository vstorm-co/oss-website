> **TL;DR:** Agenty AI nie mają wbudowanej świadomości zużycia własnego okna kontekstu — ta informacja żyje w warstwie orkiestracji, nie w historii wiadomości. `LimitWarnerCapability` w pydantic-deep v0.3.8 wypełnia tę lukę, wstrzykując wiadomości użytkownika przy 70% (URGENT) i 85% (CRITICAL), dając modelowi czas na zakończenie pracy przed automatyczną kompresją przy 90%. Również w v0.3.8: wyszukiwanie historii oparte na BM25 bez zewnętrznych zależności, `EvictionCapability` przechwytująca duże wyniki narzędzi przed wejściem do historii, rozszerzone wykrywanie plików kontekstu (7 typów plików).

W poniedziałek pokazałem, jak agenty marnują tokeny, utykając w pętlach — powtarzając to samo wywołanie narzędzia dziesiątki razy. Dziś — cichszy problem, który kosztuje tyle samo i jest znacznie trudniejszy do wykrycia.

Twój agent AI ślepnie. Nie miał pojęcia, że okno kontekstu jest w 90% pełne.

## Problem: Dwie różne rzeczywistości

Gdy uruchamiasz długie zadanie agenta, widzisz: pasek stanu pokazujący "Kontekst: 87% użyty." Jest tam, w TUI. Widzisz, że agentowi prawie kończy się miejsce.

Ale model nie widzi paska stanu. Nie ma pojęcia.

Z perspektywy modelu każda napisana wiadomość, każde wywołanie narzędzia, każdy plan — wszystko trwa normalnie. Model nie ma sygnału, że historia konwersacji się wypełnia. Nadal generuje długie odpowiedzi, inicjuje wieloetapowe plany, wywołuje narzędzia produkujące strony wyników.

Potem przy 90%: uruchamia się automatyczna kompresja. Pamięć robocza modelu zostaje skompresowana. Traci wątek tego, co zrobił 40 wiadomości temu. Zaczyna sprzeczać się z wcześniejszymi decyzjami.

To jest ślepota okna kontekstu: luka między tym, co widzi użytkownik, a tym, co model wie o własnej sytuacji.

## Dlaczego tak się dzieje

Nowoczesne interfejsy LLM pokazują zużycie kontekstu w UI. Claude Code pokazuje to w pasku stanu. Są zaprojektowane dla ludzkiego operatora.

Ale sam model — ten który generuje wyniki — widzi tylko historię wiadomości. Nie ma wbudowanej świadomości, ile okna kontekstu ta konwersacja zajmuje. Ta informacja istnieje poza konwersacją, w warstwie orkiestracji.

## Rozwiązanie: LimitWarnerCapability

W [pydantic-deep v0.3.8](https://github.com/vstorm-co/pydantic-deep) dodaliśmy `LimitWarnerCapability`. Rozwiązanie jest proste: wstrzykuj informacje o zużyciu kontekstu bezpośrednio do konwersacji jako wiadomość użytkownika, przy dwóch progach.

**Przy ~70% użycia (URGENT):**

```
You are approaching the context limit. You have used approximately 70% of your available context window. Begin wrapping up your current task. Avoid starting new complex subtasks.
```

**Przy ~85% użycia (CRITICAL):**

```
CRITICAL: Your context window is almost full. Stop current work and use /compact NOW to compress history before continuing. Do not start new tasks.
```

Nie są to modyfikacje system promptu. Są wstrzykiwane jako wiadomości użytkownika — ten sam typ wiadomości, który model traktuje jako autorytatywne dane wejściowe.

Możliwość jest automatycznie włączona gdy `context_manager=True` (domyślne w pydantic-deep):

```python
from pydantic_deep import DeepAgent

agent = DeepAgent(
    model="claude-opus-4-5",
    context_manager=True,  # domyślne — włącza LimitWarnerCapability
)
```

## Dlaczego nie system prompt?

Instrukcje w system promptcie ("jeśli zauważysz, że kończy ci się kontekst, zakończ zadanie") nie działają dobrze — model nie może rzetelnie oszacować własnego zużycia tokenów z wnętrza konwersacji.

Wstrzykiwanie w czasie rzeczywistym przy konkretnych progach działa, bo używa rzeczywistych danych z warstwy orkiestracji.

## Wyszukiwanie historii BM25: Zero zależności, czyste Python

Przepisaliśmy też `search_conversation_history`:

```python
# Przed v0.3.8: naiwne dopasowanie podciągów
results = [msg for msg in history if query in msg.content]

# Po v0.3.8: wyniki rankowane przez BM25
results = search_conversation_history(history, "explain the authentication flow")
# Rzadkie terminy (wysoki IDF) rankują wyżej
# Zapytania wielowyrazowe tokenizowane osobno
# Zero zewnętrznych zależności — czysty Python
```

Implementujemy to bez zewnętrznych zależności — czysty Python, standardowa formuła BM25 Elasticsearch/Lucene.

## EvictionCapability

`EvictionCapability` przechwytuje duże wyniki narzędzi przez hook `after_tool_execute` — **zanim** wejdą do historii wiadomości. Zapobiegasz puchnięciu kontekstu zamiast reagować na nie.

## Kluczowe wnioski

- **Ślepota okna kontekstu** jest realna: modele nie mają wbudowanej świadomości własnego zużycia kontekstu.
- **LimitWarnerCapability** wstrzykuje wiadomości użytkownika przy 70% i 85% — przed automatyczną kompresją przy 90%.
- **Wyszukiwanie BM25** zastępuje naiwne dopasowanie podciągów, zero zewnętrznych zależności.
- **EvictionCapability** przechwytuje duże wyniki narzędzi przed wejściem do historii.
- Wszystko automatyczne z `context_manager=True` (domyślne).
