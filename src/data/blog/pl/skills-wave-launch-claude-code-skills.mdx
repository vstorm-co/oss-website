78% firm z Fortune 500 wdraża asystentów AI do kodowania. 45% wygenerowanego kodu trafia na produkcję z lukami bezpieczeństwa. Po stronie contentu 76% czytelników rozpoznaje tekst pisany przez AI w ciągu trzech sekund, a engagement spada o około 47% gdy go rozpoznają.

Te trzy liczby opisują ten sam problem z dwóch stron: **outputy AI potrzebują guardraili, niezależnie czy wyjściem jest kod czy pisanie.** Dziś wydajemy dwa pakiety Claude Code skills, które siadają na tych guardrailach.

**Claude Code skill to spakowany slash-command rozszerzający twojego agenta AI o konkretną ekspertyzę, zapisany w pliku `SKILL.md` i wywoływalny z każdego runtime'u zgodnego z AGENTS.md.** Skills Wave to dwa takie pakiety, wydane tego samego dnia, ponieważ failure-mode po obu stronach workflow AI zasługuje na to samo rozwiązanie.

## production-stack-skills: 10 Claude Code skills dla produkcyjnego kodu AI

**production-stack-skills to pakiet 10 skilli audytujący kod generowany przez AI w sześciu ważonych kategoriach i zwracający wynik 0 do 100 wraz z priorytetowym action planem.**

Flagowe polecenie to `/production check`. Wskazujesz je na repo, skill czyta routes FastAPI, migracje Postgresa, Dockerfile i config, a następnie zwraca:

- Nagłówek z wynikiem (np. "Production Readiness: 34/100")
- Sześć wyników kategorii: security, error handling, observability, deployment, data layer, code quality
- Sekcję Quick Wins z deltą punktów
- Action Plan posortowany po ważonym wpływie

W pierwszych wewnętrznych uruchomieniach na klienckich repo w Vstorm sam Quick Wins bumpuje wynik o około 30 punktów. To jest liczba, którą zacytowałbym CTO pytającemu "co daje mi włączenie tego na jeden poranek".

Pozostałe dziewięć skilli dzieli obszar: `/production review`, `/production planner`, `/production fastapi`, `/production postgres`, `/production docker`, `/production deploy`, `/production monitoring`, `/production security` i `/production error-handling`. Każdy to skupiony slash-command, nie tryb monolitycznego agenta.

## content-skills: 14 brand-first skilli, które zabijają AI slop

**content-skills to pakiet 14 skilli z katalogiem `/brand/` w centrum, więc każdy kawałek contentu generowany po pięciominutowym brand-interview czyta twój BRAND.md, VOICE.md, VISUAL.md i voice samples przed napisaniem pierwszego słowa.**

Setup jest punktem wejścia. Odpalasz `/content setup` raz. Pięć pytań o firmę, voice i identyfikację wizualną. Skill zapisuje katalog `/brand/`. Od tego momentu każdy skill w pakiecie otwiera `/brand/` przy każdym wywołaniu.

Audit to punkt wyjścia. Odpalasz `/content audit` na dowolnym kawałku contentu i dostajesz wynik 0 do 100 na voice consistency, markerach anti-slop, spójności wizualnej i brand alignment.

Między setup a audit siedzi 12 skilli produkcyjnych: strategy, calendar, blog, twitter, linkedin, reddit, hackernews, presentation, infographic, image, video, repurpose. Każdy auto-źródłuje się z `/brand/`. Różnica między postem brzmiącym "Let's dive into the transformative landscape of AI" a postem brzmiącym jak twoja rzeczywista firma to różnica między content-skills czytającym BRAND.md raz przed pisaniem a LLM zgadującym.

## Jedna instalacja, dual-CLI, bezpieczny uninstall

Oba pakiety instalują się tak samo:

```bash
curl -fsSL https://raw.githubusercontent.com/vstorm-co/production-stack-skills/main/install.sh | bash
curl -fsSL https://raw.githubusercontent.com/vstorm-co/content-skills/main/install.sh | bash
```

Każda instalacja mirroruje skille do `~/.claude/` (Claude Code) oraz `~/.agents/` (Codex, Amp i wszystko zgodne z AGENTS.md). Nie wybierasz runtime'u z góry. Skill napisany dla Claude Code działa identycznie w Codex. To ma znaczenie, bo rynek Claude Code się fragmentuje, a wybór runtime-locked skilli dziś oznacza przepisywanie ich za pół roku.

Uninstall jest nudny z założenia. `/content-skills uninstall` usuwa skille. Twój `/brand/` zostaje. To twoje, nie nasze.

Oba repo są MIT.

## Dlaczego dwa pakiety tego samego dnia

**W 30+ produkcyjnych wdrożeniach AI agentów w Vstorm awarie układają się w dwa kształty: kod, który przechodzi demo, ale pada przy pierwszym produkcyjnym incydencie, oraz content, który brzmi jak napisany przez AI, bo został napisany przez AI.**

Jeden pakiet adresuje pierwsze. Drugi adresuje drugie. Oba używają tej samej architektury (pliki SKILL.md, dual-CLI, katalogi config w stylu `/brand/`, wyniki numeryczne gdzie to możliwe), bo lekcja że skille biją monolityczne agenty stosuje się do obu. Nie chcesz "jednego AI, które robi wszystko". Chcesz 24 małych, komponowalnych, audytowalnych slash-commands.

## Najczęściej zadawane pytania

### Czym są Claude Code skills i jak je zainstalować?

Claude Code skills to spakowane slash-commands oparte o plik `SKILL.md`, które rozszerzają agenta o konkretną ekspertyzę. Pakiet instalujesz jednym `curl`, który mirroruje pliki do `~/.claude/` i `~/.agents/`. Po instalacji slash-commands są dostępne natychmiast, bez restartu.

### Jaka jest różnica między production-stack-skills a content-skills?

production-stack-skills audytuje i hartuje kod generowany przez AI w kategoriach security, error handling, observability, deployment, data, code quality, zwracając wynik 0-100. content-skills audytuje i produkuje content on-brand używając katalogu `/brand/` skonfigurowanego raz, zwracając wyniki voice-consistency i anti-slop.

### Czy te skille działają w Codex, czy tylko w Claude Code?

W obu. Każdy skrypt instalacyjny mirroruje pliki do `~/.claude/` oraz `~/.agents/`, więc ten sam skill działa w Claude Code, Codex, Amp i każdym runtime zgodnym z AGENTS.md bez modyfikacji.

### Jak długo trwa `/production check` na prawdziwym repo?

Na typowym FastAPI + Postgres o kilku tysiącach linii około minuty. Sekcja Quick Wins na górze to to, co wdrażasz najpierw, zwykle do pięciu minut.

### Kiedy używać Claude Code skill zamiast pełnego agenta?

Skilla używaj gdy zadanie to ograniczona ekspertyza wywoływana jawnie. Agenta gdy zadanie jest open-ended, wieloetapowe i wymaga planowania między narzędziami. Skille to komponowalne klocki; agenty je orkiestrują.
