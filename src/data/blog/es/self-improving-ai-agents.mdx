> **TL;DR:** Los agentes de IA repiten los mismos errores entre sesiones porque cada conversacion empieza desde cero. El comando `/improve` en pydantic-deep analiza sesiones pasadas, extrae 7 tipos de insights estructurados (AgentLearning, Failure, Pattern, UserFact, Preference, Context, Decision) y propone automaticamente actualizaciones a los archivos de contexto (SOUL.md, AGENTS.md, MEMORY.md). Sin fine-tuning, sin RAG — solo extraccion estructurada de lo que el agente ya experimento, alimentado de vuelta al contexto. Cada error repetido cuesta creditos de API y tiempo de ingenieria. /improve convierte los fallos en mejoras permanentes.

Ejecutas tu agente de IA el lunes. Busca archivos de configuracion en el directorio raiz del proyecto en lugar del directorio `config/`. Lo corriges. Funciona.

Ejecutas el mismo agente el martes. Vuelve a buscar la configuracion en el directorio raiz. Lo corriges otra vez. Funciona otra vez.

Lo ejecutas el miercoles. El mismo error. La misma correccion. Los mismos tokens y tiempo desperdiciados.

Esto no es un problema de inteligencia del modelo. GPT-4.1, Claude, Gemini — todos lo hacen. Cada sesion empieza desde cero. La ventana de contexto se llena, la sesion termina y todo lo que el agente aprendio desaparece.

Despues de implementar mas de 30 agentes de IA en produccion en [Vstorm](https://vstorm.co), dejamos de aceptar esto como normal. El patron era consistente: los agentes repiten los mismos modos de fallo entre sesiones porque no tienen mecanismo para aprender de experiencias pasadas.

Asi que construimos `/improve`.

## El problema: agentes sin estado en un mundo con estado

Seamos precisos sobre lo que significa "aprendizaje de agentes". No estoy hablando de fine-tuning, RLHF o bucles de entrenamiento. Estoy hablando de algo mas simple y practico: **un agente que no repite los mismos errores**.

Los frameworks de agentes actuales — [Pydantic AI](https://ai.pydantic.dev/), LangChain, CrewAI, AutoGen — tratan las sesiones como eventos independientes. El agente tiene un system prompt, quiza algunos archivos de contexto y un historial de conversacion que se reinicia cada sesion. Si el agente descubre que tus tests estan en `tests/` (no `test/`), que tu comando de build es `make build` (no `npm run build`), o que prefieres tabs sobre espacios — ese conocimiento muere con la sesion.

El parche que usan la mayoria de equipos: actualizar manualmente el system prompt o los archivos de contexto. Despues de ver al agente luchar, un desarrollador abre `CLAUDE.md` o equivalente y agrega una nota. Funciona. Pero no escala.

Considera lo que sucede a lo largo de 50 sesiones:

- **Sesion 12:** El agente descubre que `docker-compose.yml` esta en `infra/`, no en la raiz del proyecto
- **Sesion 23:** El agente aprende que la herramienta de migracion de base de datos es Alembic, no SQL directo
- **Sesion 31:** El agente descubre que el pipeline de CI ejecuta `pytest -x --tb=short`
- **Sesion 47:** El agente descubre que el monorepo usa dependencias a nivel de workspace

Cada uno de estos descubrimientos cuesta tokens, tiempo y paciencia humana. Y son cosas que el agente ya "sabia" en algun momento — solo que en una sesion diferente que ya no existe.

## Presentamos /improve: analisis automatizado de sesiones

`/improve` es un comando en [pydantic-deep](https://github.com/vstorm-co/pydantic-deep), nuestro framework open-source de agentes de IA que implementa el "deep agent pattern" (la misma arquitectura usada por Claude Code, Manus y Devin).

A alto nivel, `/improve` hace tres cosas:

1. **Analiza sesiones pasadas** — lee los logs de conversacion de los ultimos N dias
2. **Extrae insights estructurados** — identifica 7 tipos de aprendizajes de esas sesiones
3. **Actualiza archivos de contexto** — propone cambios en SOUL.md, AGENTS.md y MEMORY.md

El resultado: el agente se vuelve mas inteligente despues de cada lote de sesiones, sin que nadie edite archivos manualmente.

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)
report = await analyzer.analyze(days=7)
# report.proposed_changes → automatic MEMORY.md updates
```

Un comando. Siete dias de sesiones analizadas. Archivos de contexto actualizados.

## 7 tipos de insights

El nucleo de `/improve` es su sistema de extraccion de insights. En lugar de volcar datos brutos de sesion en un prompt, categoriza los descubrimientos en 7 tipos estructurados:

**1. AgentLearning** — cosas que el agente descubrio por prueba y error:

```python
@dataclass
class AgentLearningInsight:
    learning: str     # e.g. "tests are in tests/ dir"
    category: str     # tool_chain, file_location, build_command...
    evidence: str     # Tool calls that proved this
    confidence: float # 0.0-1.0
```

_Ejemplo: "El proyecto usa `make test`, no `pytest` directamente. Confianza: 0.95 (observado en 4 sesiones)."_

**2. Failure** — que salio mal y como se resolvio:

```python
@dataclass
class FailureInsight:
    description: str  # What went wrong
    root_cause: str   # Why it happened
    resolution: str   # How it was fixed
```

_Ejemplo: "El agente intento editar `/src/main.py` pero la ruta real es `/app/main.py`. Causa raiz: asumio estructura estandar de proyecto Python. Resolucion: siempre verificar el arbol de archivos primero."_

**3. Pattern** — flujos de trabajo recurrentes del agente:

```python
@dataclass
class PatternInsight:
    pattern: str      # e.g. "read -> edit -> test -> commit"
    frequency: int    # How many times observed
    context: str      # When this pattern applies
```

_Ejemplo: "Leer archivo -> editar -> ejecutar tests -> commit. Frecuencia: 12. Contexto: todas las tareas de modificacion de codigo."_

**4. UserFact** — cosas que el agente aprendio sobre las preferencias y entorno del usuario.

**5. Preference** — como el usuario prefiere que se hagan las cosas (estilo de codigo, estilo de comunicacion, eleccion de herramientas).

**6. Context** — hechos especificos del proyecto (stack tecnologico, decisiones de arquitectura, convenciones del equipo).

**7. Decision** — decisiones clave tomadas durante las sesiones que deben persistir.

Cada tipo de insight tiene una estructura especifica, requisitos de evidencia especificos y un archivo destino especifico (SOUL.md, AGENTS.md o MEMORY.md).

## Por que esto importa: el costo de la repeticion

El caso de negocio para agentes que se auto-mejoran es directo:

**Desperdicio de creditos API.** Cada error repetido quema tokens. Un agente que gasta 5 llamadas a herramientas descubriendo tu estructura de archivos — en cada sesion — puede consumir 2,000-3,000 tokens extra por sesion. A lo largo de 50 sesiones por semana, son 100K-150K tokens desperdiciados. Con precios de GPT-4.1 o Claude, se acumula.

**Desperdicio de tiempo de ingenieria.** Cada vez que un ingeniero observa un error conocido del agente y lo corrige manualmente, es overhead por cambio de contexto. Multiplica por el numero de desarrolladores en tu equipo usando agentes de IA diariamente.

**Conocimiento acumulativo.** Lo inverso de errores acumulativos es conocimiento acumulativo. Un agente que aprende de 50 sesiones tiene contexto que tomaria horas documentar manualmente. Y a diferencia de la documentacion manual, `/improve` captura cosas que los humanos no pensarian en escribir — peculiaridades de cadenas de herramientas, patrones de error, preferencias implicitas.

## Lo que /improve NO es

Para dejar claros los limites:

- **No es fine-tuning.** Los pesos del modelo no cambian. /improve actualiza archivos de contexto que se cargan en el system prompt.
- **No es RAG.** No hay base de datos vectorial ni busqueda semantica. Los insights son estructurados, categorizados y escritos en archivos especificos.
- **No es magia.** Si tu agente fundamentalmente no puede hacer una tarea, /improve no ayudara. Optimiza lo que el agente ya sabe hacer.

Piensalo como la diferencia entre un desarrollador que mantiene un wiki personal de lecciones aprendidas vs. uno que empieza de cero cada manana. Mismas habilidades, efectividad drasticamente diferente.

## Como empezar

pydantic-deep es open source:

```bash
pip install pydantic-deep
```

```python
from pydantic_deep.improve.analyzer import ImprovementAnalyzer
from pydantic_deep.config import get_sessions_dir
from pathlib import Path

analyzer = ImprovementAnalyzer(
    model="anthropic:claude-opus-4-6",
    sessions_dir=get_sessions_dir(),
    working_dir=Path("."),
)

# Analyze last 7 days of sessions
report = await analyzer.analyze(days=7)

# Review proposed changes
for change in report.proposed_changes:
    print(f"File: {change.target_file}")
    print(f"Action: {change.action}")
    print(f"Content: {change.content}")
```

## Conclusiones clave

- **Los agentes de IA son sin estado por defecto.** Cada sesion empieza desde cero, incluso si el agente resolvio el mismo problema ayer.
- **La auto-mejora no requiere fine-tuning.** Extraccion estructurada de insights de sesiones pasadas, alimentada de vuelta en archivos de contexto, es suficiente para mejoras significativas.
- **7 tipos de insights cubren el espectro completo de aprendizaje:** fallos, patrones, aprendizajes, hechos del usuario, preferencias, contexto y decisiones.
- **El costo de la repeticion se acumula.** Tokens desperdiciados + tiempo de ingenieria desperdiciado, cada sesion, cada dia.
- **/improve automatiza lo que los buenos ingenieros hacen manualmente:** revisar trabajo pasado y escribir lecciones aprendidas.
