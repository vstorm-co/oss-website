Pydantic AI acaba de lanzar el cambio de API mas grande desde su lanzamiento. Capabilities, hooks y agent specs llegaron en v1.71+ y cambian fundamentalmente como se extienden los agentes.

Mantenemos 5 bibliotecas open-source construidas sobre [Pydantic AI](https://ai.pydantic.dev/): pydantic-ai-shields (anteriormente pydantic-ai-middleware), pydantic-ai-subagents, pydantic-ai-summarization, pydantic-ai-backend y el [full-stack AI agent template](https://github.com/vstorm-co/full-stack-fastapi-nextjs-llm-template). Las cinco han sido migradas. Este articulo cubre que cambio, por que es importante, y muestra codigo real antes/despues de nuestros repositorios.

## Que son las Capabilities?

Las capabilities son unidades reutilizables y componibles de comportamiento del agente. En lugar de pasar multiples argumentos de configuracion por separado -- tools aqui, instrucciones alla, configuracion del modelo en otro lado -- una capability empaqueta todo en un unico parametro `capabilities` del constructor del `Agent`.

Cada capability puede proporcionar:

- **Tools** (via `get_toolset()`)
- **Instrucciones** (strings estaticos o callables dinamicos)
- **Configuracion del modelo** (configuracion por paso)
- **Lifecycle hooks** (patrones before/after/wrap para runs, requests al modelo, llamadas a tools)
- **Preparacion de tools** (filtrar o modificar definiciones de tools por paso)

La clase base es `AbstractCapability`. Heredas de ella, sobreescribes los metodos que necesitas y pasas instancias al agente:

```python
from pydantic_ai import Agent
from pydantic_ai.capabilities import AbstractCapability

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[MyCapability(), AnotherCapability()],
)
```

Multiples capabilities se componen automaticamente. Los before-hooks se ejecutan en orden (cap1 luego cap2), los after-hooks en orden inverso (cap2 luego cap1), y los wrap-hooks se anidan como capas de middleware.

## Que son los Hooks?

Los hooks son los puntos de intercepcion del ciclo de vida dentro de las capabilities. Pydantic AI proporciona hooks en cuatro niveles:

1. **Run hooks** -- `before_run`, `wrap_run`, `after_run`, `on_run_error`
2. **Node hooks** -- `before_node_run`, `wrap_node_run`, `after_node_run`, `on_node_run_error`
3. **Model request hooks** -- `before_model_request`, `wrap_model_request`, `after_model_request`, `on_model_request_error`
4. **Tool hooks** -- divididos en validacion y ejecucion, cada uno con variantes before/wrap/after/error

Mas `prepare_tools` para filtrar la visibilidad de tools por paso.

Son aproximadamente 20 puntos de hook en 4 niveles del ciclo de vida. Los error hooks usan un patron elegante: **raise para propagar, return para recuperar**.

Para casos simples, la capability `Hooks` ofrece registro basado en decoradores sin subclassing:

```python
from pydantic_ai.capabilities import Hooks

hooks = Hooks()

@hooks.on.before_model_request
async def log_request(ctx, request_context):
    print(f"Enviando {len(request_context.messages)} mensajes")
    return request_context

agent = Agent("openai:gpt-4.1", capabilities=[hooks])
```

## Que son los Agent Specs?

Los agent specs separan la configuracion del agente del codigo completamente. Defines tu agente en YAML o JSON:

```yaml
model: anthropic:claude-opus-4-6
instructions: You are a helpful assistant.
capabilities:
  - WebSearch
  - Thinking:
      effort: high
```

Luego lo cargas:

```python
agent = Agent.from_file("agent.yaml")
```

Las capabilities que implementan `get_serialization_name()` y `from_spec()` estan disponibles automaticamente. Tus capabilities personalizadas tambien pueden ser controladas por YAML.

## Como migraron nuestras bibliotecas

### pydantic-ai-middleware a pydantic-ai-shields

Este fue el cambio mas dramatico. Nuestra biblioteca de middleware habia crecido para incluir `MiddlewareAgent`, `MiddlewareChain`, `ParallelMiddleware`, `ConditionalMiddleware`, `PipelineSpec`, cargadores de configuracion, un compilador -- toda una capa de abstraccion paralela.

**Lo eliminamos todo.**

La version v0.3.0 renombro el paquete a `pydantic-ai-shields` y reconstruyo todo como capabilities.

**Antes (era middleware):**

```python
from pydantic_ai_middleware import MiddlewareAgent, CostTrackingMiddleware

middleware_agent = MiddlewareAgent(
    agent,
    middlewares=[CostTrackingMiddleware(budget_limit_usd=5.0)]
)
result = await middleware_agent.run("Hello")
```

**Despues (era capabilities):**

```python
from pydantic_ai import Agent
from pydantic_ai_shields import CostTracking, PromptInjection, PiiDetector

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[
        CostTracking(budget_usd=5.0),
        PromptInjection(sensitivity="high"),
        PiiDetector(detect=["email", "ssn", "credit_card"]),
    ],
)
result = await agent.run("Hello")
```

Sin agente wrapper. Sin cadena de middleware. Los shields son simplemente capabilities que se enganchan en `before_run`, `after_run`, `prepare_tools` y `before_tool_execute` segun sea necesario.

El nuevo paquete incluye 10 capabilities: `CostTracking`, `ToolGuard`, `InputGuard`, `OutputGuard`, `AsyncGuardrail` para infraestructura, mas `PromptInjection`, `PiiDetector`, `SecretRedaction`, `BlockedKeywords` y `NoRefusals` como shields de contenido sin dependencias externas.

### pydantic-ai-subagents

La biblioteca de subagentes ahora expone `SubAgentCapability` que empaqueta el toolset y las instrucciones dinamicas:

```python
from pydantic_ai import Agent
from subagents_pydantic_ai import SubAgentCapability, SubAgentConfig

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[SubAgentCapability(
        subagents=[
            SubAgentConfig(
                name="researcher",
                description="Researches topics",
                instructions="You are a research assistant.",
            ),
        ],
    )],
)
```

### pydantic-ai-summarization

Cuatro capabilities reemplazan la antigua gestion de contexto basada en middleware:

- **`SummarizationCapability`** -- activa la sumarizacion LLM cuando se superan los umbrales
- **`SlidingWindowCapability`** -- alternativa sin costo que descarta los mensajes mas antiguos
- **`LimitWarnerCapability`** -- inyecta advertencias cuando los limites se acercan
- **`ContextManagerCapability`** -- paquete completo: seguimiento de tokens, auto-compresion, truncamiento de outputs de tools

```python
from pydantic_ai import Agent
from pydantic_ai_summarization.capability import ContextManagerCapability

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[ContextManagerCapability(max_tokens=100_000)],
)
```

### pydantic-ai-backend

Nuestro toolkit de sistema de archivos se convirtio en `ConsoleCapability` -- empaquetando tools, instrucciones y control de permisos:

```python
from pydantic_ai import Agent
from pydantic_ai_backends import ConsoleCapability
from pydantic_ai_backends.permissions import READONLY_RULESET

agent = Agent(
    "openai:gpt-4.1",
    capabilities=[ConsoleCapability(permissions=READONLY_RULESET)],
)
```

## Conclusiones clave

**1. Eliminen sus capas de abstraccion.** Eliminamos miles de lineas de codigo middleware. El framework lo hace mejor.

**2. La composicion es gratuita.** Multiples capabilities se apilan sin escribir logica de fusion.

**3. Los agent specs cambian el deployment.** Definir comportamiento del agente en YAML, desplegar cambiando un archivo de configuracion.

**4. La migracion es mecanica.** Para cada hook de middleware hay un equivalente directo en capabilities.

**5. Piensen en capabilities, no en agentes.** Patron antiguo: construir un agente especializado. Patron nuevo: construir una capability, adjuntarla a cualquier agente.
