> **TL;DR:** pydantic-deep v0.3.5 trae `--sandbox docker` — una bandera que mantiene la interfaz del agente en tu terminal mientras todas las operaciones de archivos y comandos de shell se ejecutan dentro de un contenedor Docker con tu proyecto montado en `/workspace`. Los espacios de trabajo con nombre (`--workspace <name>`) proporcionan entornos Docker persistentes entre sesiones. El runner headless (`pydantic-deep run`) añade `--json`, `--max-turns` y `--timeout` para integración CI/CD. Sin Dockerfile que escribir. El contenedor se limpia automáticamente al salir.

El miércoles redujimos la instalación a un solo comando. Ahora hablemos de dónde ejecuta realmente el agente.

Cuando ejecutas un agente de IA localmente, tiene acceso completo a tu entorno host. Puede modificar archivos, instalar paquetes, cambiar la configuración del sistema. En desarrollo está bien — quieres que el agente tenga acceso. Pero en CI/CD, o al ejecutar tareas no confiables, eso es un problema.

pydantic-deep v0.3.5 — el runtime modular de agentes para Python — trae el modo Docker sandbox. Aquí está qué construimos, por qué lo construimos así y cómo usarlo en un pipeline CI/CD real.

## Por qué importa la ejecución aislada

El riesgo con agentes en el host es sutil. No se trata de que el agente haga algo malicioso — sino de que puede hacer cosas que no pretendías. Instalar un paquete que entra en conflicto con las dependencias de tu proyecto. Modificar un archivo de configuración mientras depuras. Dejar estado que rompe el siguiente pipeline.

La solución es el aislamiento. El agente debería ver el proyecto, pero sus efectos secundarios no deberían filtrarse al host.

## Arquitectura del Docker Sandbox

En pydantic-deep, la bandera `--sandbox docker` cambia dónde ocurre la ejecución, no dónde vive la interfaz.

- La TUI y el runner headless permanecen en tu terminal como de costumbre
- Las operaciones de archivos (`read_file`, `write_file`, `list_dir`) se ejecutan dentro del contenedor Docker
- Los comandos de shell (`run_command`) se ejecutan dentro del contenedor
- Tu directorio de proyecto está montado en `/workspace` (lectura-escritura)

```bash
pydantic-deep run "analyze this codebase and suggest improvements" \
  --sandbox docker
```

Sin Dockerfile que escribir. Sin `docker run` manual. Configura la imagen sandbox en `.pydantic-deep/config.toml`:

```toml
[sandbox]
sandbox = "docker"
sandbox_image = "python:3.12-slim"
```

## Espacios de Trabajo con Nombre: Entornos Docker Persistentes

Los espacios de trabajo con nombre resuelven el problema del estado — los paquetes instalados dentro persisten entre sesiones:

```bash
pydantic-deep tui --workspace ml-env
# los paquetes instalados persisten entre sesiones
# múltiples hilos comparten el mismo workspace
```

`--workspace <name>` implica automáticamente `--sandbox docker` — no necesitas ambas banderas.

Gestión de espacios de trabajo:

```bash
# Todos los workspaces con estado, imagen, fecha de creación
pydantic-deep sandbox list

# Detener un workspace específico
pydantic-deep sandbox stop ml-env

# Detener y eliminar todos los workspaces del proyecto
pydantic-deep sandbox stop all --rm
```

## Runner Headless para CI/CD

```bash
pydantic-deep run "run all tests and fix failures" \
  --sandbox docker \
  --max-turns 50 \
  --timeout 300 \
  --json
```

Banderas clave para CI/CD:

| Bandera | Propósito |
|---------|-----------|
| `--json` | Salida legible por máquina para parseo en pipelines |
| `--max-turns` | Límite de iteraciones del agente (previene costos desbocados) |
| `--timeout` | Timeout wall-clock en segundos |
| `--model` | Override del modelo por ejecución |
| `--no-subagents` | Deshabilita subagentes para ejecuciones deterministas |

pydantic-deep elimina el límite predeterminado de 50 requests de pydantic-ai para que las tareas largas de CI no fallen silenciosamente. Controlas los límites explícitamente mediante `--max-turns` y `--timeout`.

## Requisitos

```bash
pip install 'pydantic-ai-backend[docker]'
```

## Conclusiones Clave

- `--sandbox docker` ejecuta operaciones de archivos y shell dentro de un contenedor Docker, con tu proyecto en `/workspace`
- Los espacios de trabajo con nombre (`--workspace <name>`) preservan paquetes instalados y estado entre sesiones
- El runner headless (`pydantic-deep run`) es nativo para CI/CD: `--json`, `--max-turns`, `--timeout`, paridad completa de banderas con TUI
- La gestión de workspaces (`pydantic-deep sandbox list/stop`) proporciona visibilidad y control
- Requiere `pip install 'pydantic-ai-backend[docker]'`
