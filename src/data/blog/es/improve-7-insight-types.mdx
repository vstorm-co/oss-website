> **TL;DR:** El comando /improve en pydantic-deep ejecuta un pipeline de 4 etapas — Discovery, Extraction, Raw Traces, Synthesis — que analiza sesiones pasadas del agente y propone cambios en los archivos de contexto. La etapa de extracción divide sesiones en ventanas de 140K tokens con overlap de 5 mensajes y extrae 7 tipos de insights estructurados usando agentes Pydantic AI. La etapa de síntesis combina insights entre sesiones, puntúa la confianza y solo propone cambios por encima de 0.7. Todo el pipeline es automatizado — cero lectura manual de logs.

## El Pipeline

`/improve` no es una sola llamada a un LLM. Es un pipeline de 4 etapas orquestado por una clase `ImprovementAnalyzer`:

```python
async def analyze(self, days=7, max_sessions=20, focus=None) -> ImprovementReport:
    # 1. Discover sessions
    session_paths = self._discover_sessions(days)
    session_paths = session_paths[:max_sessions]

    # 2. Extract insights per session
    insights: list[SessionInsights] = []
    for path in session_paths:
        session_insights, chunks = await self._extractor.extract(path)
        insights.append(session_insights)

    # 3. Load current context + raw tool sequences
    current_context = self._load_current_context()
    tool_sequences = self._load_tool_sequences(session_paths)

    # 4. Synthesize
    proposed_changes = await self._synthesizer.synthesize(
        insights, current_context, tool_sequences=tool_sequences
    )
    return ImprovementReport(...)
```

Cada etapa tiene un trabajo específico. Vamos a recorrerlas.

## Etapa 1: Discovery

La etapa más simple. `/improve` escanea `~/.pydantic-deep/sessions/` buscando directorios de sesiones de los últimos N días (por defecto: 7). Cada sesión es un directorio con logs de conversación y trazas de ejecución de herramientas.

Puedes controlar la ventana: `days=7` mira la última semana, `max_sessions=20` limita el análisis. El parámetro `focus` indica al extractor en qué concentrarse.

Nada sofisticado — simple recorrido del sistema de archivos con filtrado por fecha.

## Etapa 2: Extraction — Donde ocurre la magia

Este es el núcleo de `/improve`. Un `SessionExtractor` procesa cada sesión y extrae 7 tipos de insights estructurados.

### El problema del chunking

Las sesiones de agentes pueden ser enormes. Una sesión de codificación de 2 horas supera fácilmente los 500K tokens. No puedes alimentar eso a un LLM de una sola vez.

La solución: dividir sesiones en ventanas de 140.000 tokens máximo, con overlap de 5 mensajes entre chunks para mantener la continuidad del contexto:

- **MAX_TOKENS_PER_CHUNK = 140.000** — cabe cómodamente en la mayoría de ventanas de contexto de modelos
- **OVERLAP_MESSAGES = 5** — los últimos 5 mensajes del chunk N se convierten en los primeros 5 del chunk N+1
- **Estimación de tokens** — ~4 caracteres por token (heurística rápida, suficiente para chunking)
- **Truncamiento de output de herramientas** — 70% cabeza, 30% cola (preserva tanto el inicio como el final de outputs largos)

La división 70/30 para truncamiento de output es deliberada. El inicio de un output de herramienta suele tener el comando y los resultados iniciales. El final tiene el estado final o el mensaje de error. La parte media es frecuentemente output repetitivo que se puede descartar.

### Los 7 tipos de insights

Cada chunk es analizado por un agente Pydantic AI con structured output — no texto libre, sino dataclasses tipadas:

**1. FailureInsight** — Qué salió mal, por qué y cómo se arregló.

```python
@dataclass
class FailureInsight:
    description: str  # What went wrong
    root_cause: str   # Why it happened
    resolution: str   # How it was fixed
    tool_calls: list[str] = field(default_factory=list)
```

El tipo de insight más valioso. Si tu agente falló al ejecutar tests porque olvidó activar un virtualenv, `FailureInsight` lo captura — y la síntesis puede escribir "siempre activar .venv antes de pytest" en el contexto del agente.

**2. PatternInsight** — Secuencias recurrentes de llamadas a herramientas.

```python
@dataclass
class PatternInsight:
    pattern: str      # e.g., "read → edit → test → commit"
    frequency: int
    context: str
```

Si tu agente consistentemente hace `read file → edit → run tests → commit` al arreglar bugs, es un patrón que vale la pena codificar.

**3. PreferenceInsight** — Preferencias de estilo del usuario extraídas de correcciones.

```python
@dataclass
class PreferenceInsight:
    preference: str
    evidence: str     # User correction or explicit request
```

"No uses el tipo `any` en TypeScript" — capturado de una sesión donde corregiste al agente.

**4. ContextInsight** — Hechos arquitectónicos sobre el proyecto.

```python
@dataclass
class ContextInsight:
    fact: str
    confidence: float
```

"El proyecto usa Alembic para migraciones de base de datos" — descubierto de tool traces mostrando comandos `alembic upgrade head`.

**5. DecisionInsight** — Decisiones importantes tomadas durante sesiones.

```python
@dataclass
class DecisionInsight:
    decision: str
    reasoning: str
    confirmed: bool
```

"Cambio de REST a WebSocket para streaming de respuestas" — con el razonamiento. La flag `confirmed` indica si el usuario aprobó explícitamente.

**6. UserFactInsight** — Hechos sobre ti, el desarrollador.

```python
@dataclass
class UserFactInsight:
    fact: str
    category: str     # identity, role, language, expertise, preference
    confidence: float
```

Categorías: identity, role, language, expertise, preference. "El usuario prefiere comentarios en polaco pero código en inglés" — extraído de interacciones reales.

**7. AgentLearningInsight** — Cosas que el agente descubrió sobre el entorno.

```python
@dataclass
class AgentLearningInsight:
    learning: str
    category: str     # tool_chain, file_location, build_command, environment
    evidence: str
    confidence: float
```

"pytest se ejecuta desde la raíz del proyecto con `python -m pytest`" — con el tool trace real como evidencia.

## Etapa 3: Raw Tool Traces

Aquí es donde `/improve` se diferencia de enfoques más simples.

En lugar de pasar solo los insights extraídos a la síntesis, también cargamos archivos `tool_log.jsonl` sin procesar de cada sesión. Esto sigue el **principio Meta-Harness**: las trazas de ejecución sin procesar contienen información diagnóstica que se pierde cuando un LLM las resume.

Piénsalo — un LLM extrayendo insights podría anotar "el test falló." La traza sin procesar muestra _exactamente_ qué assertion falló, cuáles eran los valores esperados vs. reales y qué comando se ejecutó.

## Etapa 4: Síntesis

El sintetizador toma todo — insights extraídos de todas las sesiones, archivos de contexto actuales y trazas sin procesar — y produce cambios propuestos.

```python
agent: Agent[None, _SynthesisOutput] = Agent(
    model=self._model,
    output_type=_SynthesisOutput,
    instructions=prompt,
)
result = await agent.run(user_prompt)
```

De nuevo, structured output vía Pydantic AI. Cada cambio propuesto incluye:

- **target_file** — qué archivo actualizar (SOUL.md, AGENTS.md o MEMORY.md)
- **change_type** — create (archivo nuevo), append (añadir sección), update (reemplazar sección)
- **confidence** — 0.0 a 1.0. Solo cambios con **>= 0.7** se proponen
- **source_sessions** — qué sesiones proporcionaron evidencia

El umbral de confianza es clave. Una sola sesión con `npm run build` no significa que ese sea siempre tu comando de build. Pero si 8 de 10 sesiones lo usan — la confianza es alta y el cambio se propone.

### Aplicación de cambios

Tres tipos de cambios con semántica clara:

- **create** — Archivo nuevo (ej. primera vez generando MEMORY.md)
- **append** — Añadir contenido al final de un archivo existente
- **update** — Encontrar sección por encabezado, reemplazar contenido hasta el siguiente encabezado de igual o mayor nivel

Los cambios pasan por revisión en el TUI antes de aplicarse.

## Persistencia de estado

`/improve` rastrea su propia historia en `.pydantic-deep/improve_state.json`:

- Total de ejecuciones
- Sesiones analizadas
- Cambios aceptados vs. rechazados

Esto significa que `/improve` no re-analiza sesiones ya procesadas y aprende de tus decisiones de aceptar/rechazar.

## Por qué importa el Structured Output

Cada paso de extracción y síntesis usa agentes Pydantic AI con esquemas de output tipados. Sin parsing de regex. Sin "extraer JSON del bloque de código markdown." El modelo devuelve dataclasses tipadas, validadas en el límite.

Esto no es solo código limpio — es fiabilidad. Cuando procesas 20 sesiones con múltiples chunks cada una, una sola extracción malformada rompe todo el pipeline. Structured output vía Pydantic AI elimina ese modo de fallo.
