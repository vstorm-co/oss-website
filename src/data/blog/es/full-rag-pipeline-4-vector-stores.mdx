## La arquitectura: 5 pasos, cada uno configurable

Todo sistema RAG hace lo mismo: **parse → chunk → embed → store → search**. La diferencia está en cuántas decisiones tienes que tomar en cada paso.

En nuestro template, cada paso es una abstracción intercambiable:

```
Document Upload
  │
  ├── Parse: PyMuPDF (default) | LlamaParse (130+ formats) | python-docx
  │
  ├── Chunk: recursive (default) | markdown | fixed
  │     └── chunk_size=512, overlap=50 (configurable via env vars)
  │
  ├── Embed: OpenAI | Voyage | Gemini (multimodal) | SentenceTransformers (local)
  │     └── dimensions auto-derived from model name
  │
  ├── Store: Milvus | Qdrant | ChromaDB | pgvector
  │
  └── Search: vector | hybrid (BM25 + vector + RRF) | + reranking (Cohere | CrossEncoder)
```

Eliges tu stack durante la generación del proyecto. El template lo conecta todo. Sin código de pegamento.

## 4 bases de datos vectoriales, 1 interfaz

La mayor decisión de diseño fue hacer las bases vectoriales intercambiables. Implementamos `BaseVectorStore` con cuatro backends:

```python
class BaseVectorStore(ABC):
    async def insert_document(self, collection_name: str, document: Document) -> None
    async def search(self, collection_name: str, query: str, limit: int = 4) -> list[SearchResult]
    async def delete_document(self, collection_name: str, document_id: str) -> None
    async def get_collection_info(self, collection_name: str) -> CollectionInfo
```

**Milvus** — grado de producción, se ejecuta como 3 servicios Docker (etcd + MinIO + Milvus). Ideal para despliegues a gran escala. Cosine similarity con indexación IVF_FLAT.

**Qdrant** — un solo servicio Docker, gran equilibrio entre rendimiento y simplicidad. Nuestra recomendación por defecto para la mayoría de equipos.

**ChromaDB** — modo embebido, sin Docker necesario. Perfecto para prototipado y desarrollo local. Solo `pip install chromadb`.

**pgvector** — usa tu PostgreSQL existente. Sin nueva infraestructura. Indexación HNSW. Si ya tienes Postgres, es la opción con menor fricción.

¿Cambiar entre ellos? Una variable de entorno:

```bash
# In your .env:
VECTOR_STORE=qdrant    # or: milvus, chromadb, pgvector
```

El template gestiona automáticamente los connection strings, servicios Docker, creación de esquemas y configuración de índices.

## Búsqueda híbrida: por qué la búsqueda solo vectorial no es suficiente

La búsqueda vectorial pura funciona bien para consultas semánticas ("documentos sobre seguridad de edificios"). Falla en coincidencias exactas ("encuentra el contrato #2024-0847") porque los embeddings no preservan cadenas exactas.

Nuestra búsqueda híbrida combina ambos enfoques:

```python
async def retrieve(self, query: str, collection_name: str, limit: int = 5):
    # Step 1: Vector search (semantic)
    raw_results = await self.store.search(collection_name, query, limit=limit * fetch_multiplier)

    # Step 2: BM25 keyword search
    if self._hybrid_enabled:
        bm25_results = await self._bm25_search(query, collection_name, limit * fetch_multiplier)
        if bm25_results:
            raw_results = self._rrf_fuse(raw_results, bm25_results)

    # Step 3: Rerank (optional)
    if should_rerank and self.rerank_service:
        results = await self.rerank_service.rerank(query=query, results=raw_results, top_k=limit * 2)

    return results[:limit]
```

La fusión utiliza **Reciprocal Rank Fusion (RRF)** — un algoritmo simple pero efectivo que combina rankings de múltiples fuentes:

```python
@staticmethod
def _rrf_fuse(vector_results, bm25_results, k=60):
    scores = {}
    for rank, r in enumerate(vector_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    for rank, r in enumerate(bm25_results):
        key = r.content[:100]
        scores[key] = scores.get(key, 0) + 1.0 / (k + rank + 1)
    return sorted_by_score(scores)
```

Actívalo con una variable de entorno: `RAG_HYBRID_SEARCH=true`.

## Reranking: el multiplicador de calidad

El retrieval inicial lanza una red amplia. El reranking la estrecha. Soportamos dos opciones:

**Cohere Reranker** (API) — la forma más rápida de mejorar la calidad del retrieval. Envía tus resultados + consulta, recíbelos re-puntuados por un modelo entrenado específicamente para ranking de relevancia:

```python
response = await self.client.rerank(
    query=query,
    documents=[result.content for result in results],
    model="rerank-v3.5",
    top_n=top_k,
)
```

**CrossEncoder** (local) — ejecuta un modelo cross-encoder de SentenceTransformers localmente. Sin llamadas API, ningún dato sale de tu infraestructura:

```python
pairs = [[query, result.content] for result in results]
scores = self.model.predict(pairs)  # Runs locally on CPU/GPU
```

El pipeline es: recuperar 3x más resultados de los necesarios, rerankear, devolver top-k. Esto mejora consistentemente la precisión sin tocar tus embeddings o vector store.

## Versionado de documentos: deduplicación SHA256

Re-ingestar un documento no debería crear duplicados. Nuestro pipeline usa hashing de contenido:

```python
async def ingest_file(self, filepath, collection_name, replace=True):
    document = await self.processor.process_file(filepath)

    # Check for existing version by source path or content hash
    existing_id = await self._find_existing_by_source(collection_name, source_path)
    if not existing_id:
        existing_id = await self._find_existing_by_hash(collection_name, document.metadata.content_hash)

    # Replace old chunks with new ones
    if existing_id:
        await self.store.delete_document(collection_name, existing_id)

    await self.store.insert_document(collection_name, document)
```

¿Sincronización con Google Drive? Misma lógica — los archivos modificados se re-embeben, los sin cambios se omiten.

## 4 proveedores de embeddings

| Proveedor            | Modelo                     | Dimensiones | ¿API Key?  |
| -------------------- | -------------------------- | ----------- | ---------- |
| OpenAI               | text-embedding-3-small     | 1536        | Sí         |
| Voyage               | voyage-3                   | 1024        | Sí         |
| Gemini               | gemini-embedding-exp-03-07 | 3072        | Sí         |
| SentenceTransformers | all-MiniLM-L6-v2           | 384         | No (local) |

Las dimensiones se derivan automáticamente del nombre del modelo — sin configuración manual:

```python
EMBEDDING_DIMENSIONS = {
    "text-embedding-3-small": 1536,
    "voyage-3": 1024,
    "gemini-embedding-exp-03-07": 3072,
    "all-MiniLM-L6-v2": 384,
}
```

Gemini es el más interesante — soporta **embeddings multimodales**. Texto e imágenes en el mismo espacio vectorial. Lo usamos para la extracción de descripciones de imágenes de PDFs.

Los resultados incluyen atribución de fuente: nombre de archivo, número de página, número de chunk y puntuación de similitud. El system prompt del agente le indica citar fuentes con referencias `[1]`, `[2]`.

## Conclusiones clave

- **RAG es un pipeline de 5 decisiones** (parse, chunk, embed, store, search) — nuestro template hace cada una configurable sin cambios de código
- **La búsqueda solo vectorial pierde coincidencias exactas** — híbrida (BM25 + vector + RRF) captura tanto consultas semánticas como de palabras clave
- **Reranking es la mejora de calidad más barata** — recuperar 3x más + rerankear supera consistentemente al ajuste de embeddings
- **El versionado de documentos previene chunks duplicados** — hash SHA256 del contenido + seguimiento de ruta de origen
- **Una variable de entorno cambia todo** — `VECTOR_STORE=pgvector`, `RAG_HYBRID_SEARCH=true`, `EMBEDDING_MODEL=voyage-3`
